### 3.3. Поручение 4: Прототип элемента "Цена блюда"

#### Контекст проблемы

Ранее элемент назывался "Сумма блюда" (в задаче), но правильное название - "Цена блюда" (Руслан: "Цена блюда - это стандартная. Сумма - это то, что складывается из нескольких позиций").

Основная проблема: в конструкторе Темы (iikoWeb) невозможно показать реальную цену блюда, потому что:
- Тема создается общая, без привязки к конкретной торговой точке
- В разных точках одно и то же блюдо может стоить по-разному (180 руб. vs 200 руб.)
- Backend (Роман) не отдает цену в данных для конструктора
- Вопрос источника цены на стороне iikoWeb не решен (есть "цена в карточке" и "цена по прайсу", возможны разные источники цен)

Обсуждалось использование внешнего меню (там цена и картинка есть сразу), но решили пока не менять - работаем с классическим айковским меню.

#### Требования к прототипу

| Требование | Описание |
|-----------|---------|
| Отображение в конструкторе | Вместо реальной цены показывать тестовое/примерное значение |
| Поле ввода | Пользователь (маркетолог/настройщик) может ввести произвольное число для предпросмотра внешнего вида |
| Info-иконка | Рядом с полем - восклицательный знак (или иконка "i"). При наведении (tooltip) - пояснение: "Это не реальная цена. Значение используется только для предварительного просмотра. Реальная цена будет взята из меню вашей торговой точки" |
| Значение по умолчанию | Обсуждалось: можно поставить какое-то значение (например, "350" или знаки-заглушки). Точный формат по умолчанию - на усмотрение при прототипировании |
| Привязка к блюду | Когда элемент привязан к конкретному блюду из меню, tooltip на элементе в конструкторе должен показывать название привязанного блюда |
| Пустое/нулевое значение | Если значение = 0, это означает "не назначено" (пометить в инструкции) |

#### Отображение на реальном экране (iikoFront, плагин)

На реальном экране (не в конструкторе) должна отображаться настоящая цена блюда, полученная из iikoFront.

### 3.4. Поручение 5: Прототип элемента "Количество блюд в заказе" (Счетчик)

#### Описание

Новый элемент, отображающий количество блюд в определенных статусах относительно общего количества.

#### Требования к прототипу

| Требование | Описание |
|-----------|---------|
| Формат отображения | "X из Y" (через дробь). Например: "3 из 8" |
| X (числитель) | Количество блюд в выбранных статусах (фильтр по статусам) |
| Y (знаменатель) | Общее количество блюд в заказе |
| Фильтр по статусам | Настройка в конструкторе: чекбоксы со списком статусов. Пользователь выбирает, в каких статусах считать блюда для числителя |
| Размещение | Внутри контрола заказа (например, под списком позиций или в любом месте по выбору пользователя) |
| Назначение | Гость (или оператор) сразу видит, сколько блюд готово из общего количества, и может принять решение - идти забирать или ждать |

#### Пример использования

Заказ из 8 блюд. В настройках счетчика выбран статус "Готово". 3 блюда уже готовы. На экране отображается: **3 / 8**.

### 3.5. Поручение 6: Контрол "Состав заказа" - итоговые договоренности

#### Визуальная структура контрола

Контрол "Состав заказа" состоит из следующих визуальных частей:

1. **Подложка** - прямоугольник с белым (или светлым) фоном, скругление 25, тень. Является самым нижним слоем
2. **Шапка** - верхняя часть подложки, отдельный прямоугольник другого цвета (серый/темный), скругление 5. Содержит текст "Заказ [номер]" (из поля "Номер заказа" с префиксом)
3. **Список позиций (блюд)** - динамический список с названиями блюд
4. **Счетчик (опционально)** - элемент "Количество блюд в заказе" (см. п.3.4)

#### Настройки контрола

| Настройка | Тип | Значение по умолчанию | Описание |
|-----------|-----|----------------------|---------|
| Динамическая высота | Галочка (checkbox) | Включена | Если включена: контрол сжимается/расширяется по количеству строк в списке блюд. Если выключена: контрол фиксированного размера, отображает столько строк, сколько помещается |
| Показывать заголовок | Галочка (checkbox) | Включена | Если выключена: шапка контрола скрывается. Клиент может нарисовать свою шапку отдельными элементами |

#### Поведение динамической высоты

- Когда "Динамическая высота" включена: контрол расширяется вниз при увеличении количества позиций и сжимается при уменьшении
- Сложный кейс (отложен на будущее): когда под контролом расположены другие элементы (например, счетчик или подложка), они тоже должны сдвигаться. Это пока НЕ реализуется
- Первая версия: при динамической высоте сжимается/расширяется только сам список и его подложка. Сопутствующие элементы вне контрола НЕ двигаются
- Руслан: "Мы сейчас выпустили простой объект - это уже круто. Сложные объекты - в следующих версиях"

#### Кейс статической высоты (не тянется)

Руслан обосновал необходимость галочки: если отключить динамическую высоту, можно нарисовать компактный контрол на 2-3 позиции. В сочетании со счетчиком пользователь все равно видит, что в заказе есть еще блюда. Это допустимый и "неплохо выглядящий" вариант.

#### Перспектива: связь с menu boards

Руслан отметил, что контрол "Состав заказа" со списком и динамической высотой - это база для будущих menu boards (меню-борды). Объект menu board будет наследоваться от того же класса с немного другими свойствами. Это нужно учитывать при проектировании.

## 4. Персонализированные коды для продажи тем (информация для справки)

Руслан озвучил концепцию будущего механизма:

- iiko как вендор рисует красивые темы
- Клиент (ресторан) хочет купить тему
- При экспорте формируется приватный код, привязанный к юрлицу клиента
- Клиент предоставляет свой код (идентификатор юрлица), экспорт привязывается к этому коду
- Тема работает только у того юрлица, которому продана (нельзя скопировать другому)
- В перспективе - магазин тем в бэк-офисе

**Статус:** Концепция. Нужно согласование с руководством. Механизм будет отдельным от текущего экспорта. В текущий scope НЕ входит.
