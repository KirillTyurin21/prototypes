## **1. Общее описание задачи**

> **О чем:** Реализация механизма удаленного получения скриншотов с экранов Digital Signage (Customer Screen / Arrivals) через веб-интерфейс бэкенда. Пользователь (маркетолог/администратор) должен иметь возможность увидеть, что в данный момент отображается на конкретном экране конкретного ресторана, не заходя на фронт.

**Цель:** Дать пользователю инструмент для оперативной проверки того, что реально показывает экран после изменения настроек/тем, без необходимости физического доступа к терминалу.

**Примечание:** "Arrivals" - правильное название (не "Rivals"). Механизм общий для Customer Screen и Arrivals, реализация начинается с Customer Screen.

## **2. Бизнес-сценарии (зачем это нужно)**

### Сценарий 1: Диагностика проблемы конкретного ресторана
1. Звонят из ресторана: "У нас на экране что-то не то"
2. Пользователь заходит в настройки терминала в веб-интерфейсе
3. Нажимает "Получить скриншот" для этого ресторана
4. Видит в попапе, что реально отображается на экране
5. Сравнивает с другим рестораном - если там все ок, значит проблема НЕ в теме

### Сценарий 2: Проверка применения изменений в теме
1. Маркетолог изменил тему (картинки, контролы и т.д.)
2. Отправил настройки на ОДИН тестовый ресторан (через кнопку "Отправить настройки")
3. Запросил скриншот с этого ресторана
4. Убедился, что изменения применились корректно
5. Только после этого раскатывает настройки на всю сеть

### Сценарий 3: Мониторинг сети
1. Маркетолог подготовил акцию (например, к 8 марта)
2. Загрузил новые картинки и темы
3. Отправил на все рестораны
4. Выборочно проверяет скриншотами, что контент применился

## **3. Архитектура решения (согласована на встрече)**

### 3.1. Общая схема взаимодействия

```
[Веб-UI бэкенда]                [Транспорт]         [Плагин Digital Signage]      [Сервер картинок]
       |                              |                        |                          |
       |--- 1. Команда "Сделай ------>|--- 2. Команда -------->|                          |
       |       скриншот"              |    на фронт            |                          |
       |                              |                        |--- 3. Делает скриншот    |
       |                              |                        |       экрана             |
       |                              |                        |                          |
       |                              |                        |--- 4. Загружает -------->|
       |                              |                        |       картинку           |
       |                              |                        |       (напрямую)         |
       |                              |                        |                          |
       |                              |<-- 5. Метаданные ------|                          |
       |                              |    (ID, имя файла,     |                          |
       |<-- 6. Сигнал: "Готово" ------|     время)             |                          |
       |    + метаданные              |                        |                          |
       |                              |                        |                          |
       |--- 7. Запрашивает ---------->|----------------------->|------------------------->|
       |       картинку по ID         |                        |                          |
       |<-- 8. Картинка --------------|<-----------------------|<-------------------------|
       |                              |                        |                          |
       |--- 9. Показывает            |                        |                          |
       |       в попапе              |                        |                          |
```

### 3.2. Ключевые принципы

| Принцип | Описание |
|---------|----------|
| **Минимум данных через транспорт** | Через транспорт передаются только короткие сообщения (команды и метаданные). Сама картинка (~1 МБ) НЕ передается через транспорт |
| **Сервер картинок** | Картинка загружается напрямую с плагина на сервер картинок (тот же, что используется для галереи) |
| **Метаданные через транспорт** | Обратно через транспорт приходит: ID картинки, имя файла, время создания |
| **Запись в таблицу** | Серверная часть записывает информацию по картинке в таблицу для возможности последующей очистки |
| **Очистка** | При новом запросе скриншота - предыдущий удаляется |

## **4. Требования к прототипу (UI)**

### 4.1. Экран "Настройки терминалов" - основное рабочее место

На экране настроек терминалов (где отображается список терминалов/экранов ресторана) необходимо добавить:

#### Кнопка 1: "Отправить настройки"
- **Расположение:** В области терминала/экрана (рядом с галочками выбора)
- **Действие:** Отправляет команду на плагин - "Запроси свежие настройки с бэка"
- **Принцип работы:** Мы НЕ отправляем настройки плагину напрямую, а говорим плагину: "Запроси себе настройки". Плагин сам приходит к бэку и забирает актуальную конфигурацию
- **Галочки выбора:** Пользователь может выбрать конкретные терминалы (один или несколько) и отправить настройки только на выбранные
- **Подсветка изменений:** Если были изменения в теме/настройках, но еще не отправлены - кнопка должна визуально сигнализировать об этом (например, становиться активной/выделенной)

#### Кнопка 2: "Получить скриншот" (Посмотреть экран)
- **Расположение:** Рядом с "Отправить настройки", в области конкретного терминала
- **Действие:** Запрашивает снимок экрана с конкретного терминала в реальном времени
- **Результат:** Открывает попап с изображением

### 4.2. Попап просмотра скриншота

При получении скриншота открывается модальное окно (попап) со следующим содержимым:

| Элемент | Описание | Пример |
|---------|----------|--------|
| **Картинка** | Полученный скриншот экрана | Изображение |
| **Название ресторана** | Из какого ресторана скриншот | "Ресторан на Горке" |
| **Терминал/Экран** | Какой именно терминал или экран | "Терминал 3" / "Экран Восток" |
| **Время скриншота** | Когда был сделан снимок (по LOCAL времени ресторана) | "23:30 (Владивосток)" |
| **Разрешение экрана** | Разрешение, с которым сделан скриншот | "1920x1080" |

**Важно:** Время отображается по МЕСТНОМУ времени ресторана, чтобы администратор из Москвы понимал, что скриншот сделан во Владивостоке в 23:30 по местному времени.

### 4.3. Состояния кнопки "Получить скриншот"

| Состояние | Вид | Поведение |
|-----------|-----|-----------|
| **Готова** | Активная кнопка | Можно нажать |
| **Ожидание (Pending)** | Заблокирована + индикатор загрузки | Кнопка заблокирована. Отображается окно/оверлей "Идет получение данных..." Пользователь не может нажать повторно |
| **Получено** | Открывается попап | Показывает скриншот с метаинформацией |
| **Ошибка** | Сообщение об ошибке | Если плагин не ответил / не поддерживает функционал |

### 4.4. Защита от множественных нажатий (debounce)

- **Основной подход (согласован):** После нажатия кнопка БЛОКИРУЕТСЯ до получения ответа
- Может также отображаться модальное окно "Идет получение данных...", блокирующее любые повторные действия
- Если по какой-то причине прилетит несколько команд подряд, фронт выполняет только ПОСЛЕДНЮЮ
- Ограничение по частоте (раз в 5 минут, предложение Вани) было отклонено как слишком строгое. Вместо этого реализуется блокировка кнопки до получения ответа от сервера

## **5. Логика работы с выбором терминалов**

### 5.1. Галочки выбора терминалов

В списке терминалов/экранов ресторана уже есть колонка с информацией. Необходимо добавить:

- **Галочки (чекбоксы)** для выбора конкретных терминалов
- Возможность выбрать один, несколько или все терминалы
- Кнопка "Отправить настройки" работает для ВЫБРАННЫХ терминалов

### 5.2. Сценарий точечной проверки

1. Пользователь выбирает ОДИН тестовый ресторан/терминал (галочкой)
2. Нажимает "Отправить настройки" -> настройки уходят на этот один терминал
3. Нажимает "Получить скриншот" -> видит результат
4. Если все ок - выбирает все терминалы и отправляет настройки на всю сеть

### 5.3. Отправка на большую сеть

- При массовой отправке (сотни ресторанов) команды отправляются с задержкой (разнесение по времени), чтобы не создавать пиковую нагрузку на транспорт
- Точные параметры задержки определяются на этапе разработки

## **7. Версионирование плагина**

| Аспект | Требование |
|--------|-----------|
| **Проверка версии** | Бэкенд должен проверять версию плагина перед отправкой команды |
| **Передача версии** | Плагин уже передает свою версию в сообщениях - бэкенд начинает ее сохранять |
| **Недоступен функционал** | Если версия плагина старая (не поддерживает скриншоты), UI должен показать, что функционал недоступен для данного терминала |
| **Customer Screen (DS)** | Проще - можно раскатить новую версию плагина на всех и не заворачиваться с проверкой версий |
| **Arrivals** | Сложнее - нет автообновления, стабильная версия не найдена. Версионирование критично |

## **8. Хранилище скриншотов**

| Параметр | Описание |
|----------|----------|
| **Где хранятся** | Сервер картинок (тот же, что для галереи Digital Signage) |
| **Доступ** | Плагин загружает напрямую на сервер картинок (БЕЗ прохождения через транспорт) |
| **Аутентификация** | Для загрузки на сервер картинок могут потребоваться креды (как для галереи). Решение пока НЕ принято - вопрос безопасности вшивания кредов в плагин открыт (Михаил поднял на встрече). Требуется эксперимент: Рома проверяет, можно ли загружать без кредов |
| **Очистка** | При новом запросе предыдущий скриншот удаляется. Информация об ID/названии картинки сохраняется в таблице для механизма очистки |
| **Размер** | ~1 МБ на скриншот (~38 тысяч символов в Base64) |
