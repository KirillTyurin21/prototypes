## 1. Скоуп 1: Конструктор тем (8 минут)

**Скоуп 1 — это доработки самого конструктора.** Три задачи: группировка элементов, редактирование контрола из темы, ExternalData. Начну с самой видимой.

---

### 1.1. Группировка элементов палитры

**Суть:** палитра элементов в конструкторе тем — сейчас это простыня из 27+ элементов. Администратору тяжело искать нужный. Мы группируем элементы по семантическим категориям в аккордеон.

**Что сделано в прототипе:**
- Аккордеон с 5 категориями: Базовые, Контейнеры, Декоративные, Медиа, Информационные
- Каждая категория с иконкой и счётчиком элементов
- Каждый элемент внутри категории — с иконкой из библиотеки Front
- Первая категория раскрыта по умолчанию, остальные свёрнуты
- Плавная анимация раскрытия

**Демонстрация:** открываю `http://localhost:3000/prototype/web-screens/themes-arrivals` → редактирую тему → «Добавить элемент» → показываю аккордеон.

**Нюансы (из обсуждения с Русланом):**
- Это чисто фронтовая доработка. Backend не меняется — он только валидирует `type` при сохранении темы
- В прототипе мы показываем только элементы ТЕМЫ (8 шт.), элементы КОНТРОЛА — в другом месте, и в этой палитре их показывать не надо
- Material Icons уже подключены к проекту, новых не требуется
- Поиск по элементам и перенос модели на backend — в будущих версиях

**Статус:** ✅ Прототип готов, спецификация готова. Ждёт реализации.

---

### 1.2. Редактирование контрола из темы

**Суть:** сейчас, чтобы отредактировать контрол, администратор должен: выйти из конструктора → перейти в справочник контролов → вручную найти нужный → отредактировать → вернуться. Мы делаем быстрый переход: кнопка-карандаш прямо на контроле в теме → мгновенный переход в редактор → кнопка «Вернуться в тему».

**Что сделано в прототипе:**
- В инспекторе области (area-element-inspector) — кнопка «Редактировать контрол»
- Видна только если у области выбран контрол
- По клику — роутинг на существующий редактор контрола (`arrivals-control-editor`)
- После сохранения — «Назад» возвращает в редактор темы
- Тема автосохраняется перед переходом (чтобы не потерять контекст)

**Демонстрация:** открываю редактор темы Arrivals → кликаю на область с контролом → в правой панели кнопка «Редактировать контрол» → переход → редактирую → назад.

**Нюансы (из обсуждения с Русланом):**
- **Режим «Сохранить в теме» (изменения только для одной темы) — ОТМЕНЁН.** Руслан: «нецелесообразно, пользователь потеряет контекст изменений». Остаётся только один режим: изменения применяются к самому контролу глобально
- **Кнопка иконка-карандаш на canvas** — как в исходном прототипе на Angular 8. Мы адаптировали под Angular 16
- **Не модальное окно** — полноценный переход с сохранением контекста. Модалка была бы хуже: теряется состояние canvas
- **Справочник контролов остаётся** — это дополнительный способ навигации, не замещение

**Отличие от спеки:** в спецификации упоминался режим «Сохранить как копию». В прототипе мы его не делали — Руслан подтвердил, что достаточно текущего поведения.

**Статус:** ✅ Прототип готов, спецификация готова. Ждёт адаптации Angular 8 → 16 в продукте.

### 1.3. ExternalData — элемент «Внешние данные»

**Суть:** элемент уже есть в палитре контролов Arrivals (п. 26 из 27). Но плагин DS не читает `IOrder.ExternalData`. Задача: научить плагин читать значение по ключу и отображать на экране.

**Что в спеке:**
- Плагин при обновлении заказа читает `IOrder.ExternalData` по ключу из свойств элемента
- Ключ задаётся в конструкторе: текстовое поле «Наименование ключа»
- Один элемент = один ключ
- Используются существующие SDK-методы: `TryGetOrderExternalDataByKey`

**Особенность:** эта задача **снята с Q3**, не в приоритетах текущего спринта. Но спецификация готова — чтобы когда вернёмся, не начинать с нуля.

**Нюансы:**
- ExternalData уже используют другие плагины: Mindbox (ключ `mindbox.orderId`), OneSync (`cancelReason3pl`), Beanshe KSO
- Плагин DS только **читает** ExternalData, не пишет
- Писать будут внешние системы/плагины
- Валидации формата ключа нет — ответственность на администраторе

**Статус:** ⏸️ Снят с Q3. Спецификация готова, прототип — в минимальном объёме (элемент есть в палитре, поле ключа добавлено).

## 2. Скоуп 2: MenuBoard (10 минут)

**Скоуп 2 — самый крупный.** Это новый продукт MenuBoard — форк iikoArrivals. Две задачи: несколько Advertise и Меню-лист с ценами.

---

### 2.1. Несколько динамических областей (Advertise) на тему

**Суть:** MenuBoard — форк Arrivals в отдельный продукт. Ключевое отличие: в Arrivals одна динамическая область на тему, в MenuBoard — N областей без ограничений. Плюс привязка Campaign переносится из настроек экрана в конструктор темы.

**Что сделано в прототипе:**
- Новый раздел в боковом меню: «Доска меню» → «Темы», «Настройка экранов», «Мультиэкранность»
- Конструктор темы MenuBoard (форк Arrivals-theme-editor):
  - Элемент «Динамическая область» (вместо «Рекламный блок»)
  - Можно добавить несколько — кнопка не блокируется после первой
  - В панели свойств — **мультиселектор** рекламных кампаний (можно выбрать несколько)
  - Z-index авто-ниже QR-кода
- Страница «Настройка экранов» — без колонки «Кампании» (привязка в конструкторе)
- Палитра элементов — аккордеон с категориями (как в Скоупе 1)

**Демонстрация:** открываю `http://localhost:3000/prototype/web-screens/menuboard-themes` → создаю тему → добавляю 2 динамические области → для каждой выбираю кампании → сохраняю.

**Нюансы (из обсуждения с Русланом и Кириллом):**

- **Кампании — мультивыбор, НЕ одиночный.** Это важное исправление после встречи. В первой версии прототипа был single-select. Руслан подтвердил: «Да, внутри элемента темы — мультиплекс-селектор, потому что кампаний может быть несколько в зависимости от сроков». Мы переделали на chips/multi-select
- **Campaign — общий справочник** для всех продуктов DS (CS, Arrivals, MenuBoard). При форке НЕ создаётся отдельная таблица. Изменение Campaign влияет на ВСЕ продукты
- **Разрешения:** плагин выбирает медиа-файлы того разрешения Campaign, которое совпадает с разрешением Темы. Эта логика уже работает в Arrivals и сохраняется в MenuBoard
- **Нет миграции тем** Arrivals → MenuBoard. MenuBoard — новый продукт с собственной БД
- **Мультиэкранность** — убедились что работает, Руслан подтвердил: «сам функционал доски меню будет поддерживать и мультиэкранность, потому что экранов может быть много»

**Что в спеке, но НЕ в прототипе:**
- Backend-часть форка (БД, API) — это зона ответственности backend-команды (Роман)
- Реальная интеграция с внешним меню — используем мок-данные

**Статус:** ✅ Прототип готов. Спецификация готова. Ждёт backend-форка и фронтовой реализации.

### 2.2. Меню-лист с ценами (MenuList)

**Суть:** MenuBoard должен показывать блюда с ценами. Два новых элемента:
1. **Меню-лист (MenuList)** — элемент темы: таблица строк, каждая строка = блюдо с 8 полями
2. **Контрол меню-борда** — внутри ControlArea: карточки блюд (изображение + название + цена)

**Что сделано в прототипе:**

**Меню-лист:**
- Элемент `type: 'menulist'` в палитре, добавляется на canvas как самостоятельный элемент
- Строка блюда: иконка (40×40) | название + модификаторы + размеры | цена + граммовка
- Опционально: описание, аллергены, КБЖУ — включаются чекбоксами в инспекторе
- Выбор блюд через модальное окно: дерево категорий слева, чекбоксы справа, поиск
- Инспектор: 4 группы шрифтов (название, цена, модификаторы, описание), настройки таблицы, подсветка строк
- Настраиваемый разделитель строк, размер иконки, формат цены, позиция валюты

**Контрол меню-борда:**
- Готовый мок-контрол (id=307) в выпадающем списке контролов для ControlArea
- Карточки: изображение + название + цена

**Демонстрация:** открываю редактор темы MenuBoard → «Добавить элемент» → «Меню-лист» → «Выбрать блюда» → показываю модалку → выбираю блюда → показываю canvas → показываю инспектор свойств.

**Нюансы (ключевое архитектурное решение):**

- **MenuList и Control — РАЗДЕЛЕНЫ. НЕ объединять.** Это принципиально. Руслан: «Проблема в модификаторах. Если нам надо показывать модификаторы, то это через контролы ну п***ц как неудобно делать. Только по этой причине я их разделил.» MenuList показывает модификаторы естественно в строке. Контрол — не может. Поэтому:
  - **MenuList** = элемент темы (как Text, Image). Размещается на canvas напрямую
  - **Контрол меню-борда** = контрол внутри ControlArea (как «Состав заказа»). Размещается только внутри области

- **MenuList — НЕ контрол «Состав заказа».** Хотя визуально похожи («КДС, но не КДС»), источник данных разный:
  - «Состав заказа»: SDK (`IKitchenOrder`), реальное время
  - MenuList: кэш внешнего меню, обновление ~5 мин

- **Нет пагинации в v1.** Блюда вне высоты элемента — обрезаются. Администратор должен сам рассчитать высоту. Пагинация — в v2

- **Дубли блюд не поддерживаются.** Каждый `productId` — один раз в MenuList

- **Источник данных:** внешнее меню iiko → кэш на iikoWeb → плагин через `GetTerminalSettings`. В прототипе: `MOCK_EXTERNAL_MENU`

**Что доработано после исследования (полировка):**
- Размер иконки: 32px → настраиваемый, по умолчанию 40px (как в спеке)
- Разделитель строк: настраиваемый (solid/dashed/none + цвет)
- Граммовка: настраиваемый показ/скрытие
- Формат цены: символ валюты + позиция (до/после)
- Лейбл «Модификаторы:» — можно скрыть

**Статус:** ✅ Прототип готов (v1.26–1.28). Спецификация готова. Ждёт реализации.

## 3. Скоуп 3: Customer Screen + Подсказки (5 минут)

### 3.1. Третий режим CS — «Касса не работает»

**Суть:** когда кассовая смена не открыта, экран покупателя в неопределённом состоянии. Гость не понимает — касса работает или нет. Мы добавляем 5-й режим, который автоматически включается при закрытой смене и показывает рекламную кампанию.

**Что сделано в прототипе:**
- Редактор тем Customer Screen создан по образцу реальной страницы Web (стенд)
- В выпадающем списке режимов — 5 пунктов (добавлен «Касса не работает»)
- При выборе этого режима — появляется селектор рекламной кампании
- Preview для закрытого режима — показывает заглушку кампании вместо заказа

**Демонстрация:** открываю `http://localhost:3000/prototype/web-screens/themes-cs` → редактирую тему → переключаю режим на «Касса не работает» → выбираю кампанию → показываю preview.

**Нюансы:**
- Режим активируется **автоматически** при отсутствии открытой смены. Не вручную
- Кампания — **необязательное** поле. Можно оставить пустым — тогда просто чёрный экран
- **Только одна кампания** для режима (не мультиселектор, в отличие от MenuBoard)
- **Не «Технический перерыв»** — ручная активация кассиром не входит в scope
- Настройка индивидуального текста — через существующий механизм кампаний, не через отдельное поле

**Что в спеке, но уточнено в прототипе:**
- В спеке режим назван «третьим», хотя по факту он **пятый** (4 существующих + новый). В UI мы используем название «Касса не работает»
- Селектор кампании появляется **только** при выборе этого режима — не виден для остальных

**Статус:** ✅ Прототип готов. Спецификация готова. Ждёт реализации в плагине и на фронте.

### 3.2. Отчёт подсказок (ПОЛНАЯ ПЕРЕДЕЛКА)

**Суть:** у администратора нет инструмента оценки эффективности подсказок. Мы добавляем общий дашборд по всем подсказкам.

**Что было ДО встречи с Русланом (v1.24):**
- Кнопка «Отчёт» в каждой строке таблицы подсказок
- Отдельная панель для одной подсказки
- Колонки: Показов, Срабатываний, Конверсия

**Что стало ПОСЛЕ встречи (v1.27):**
- ❌ Убрали per-hint кнопки из таблицы
- ✅ Одна общая кнопка «Отчёт» в тулбаре страницы
- ✅ **Общий дашборд** по ВСЕМ подсказкам:
  - **Круговая диаграмма** — процентное соотношение срабатываний
  - **Таблица:** Название подсказки | Срабатывание | Продаж | Конверсия
  - **Фильтр периода:** День / Неделя / Месяц (фиксированные значения, НЕ произвольный период)
- ✅ Переименование колонок: «Показов» → «Срабатывание», «Срабатываний» → «Продаж»

**Демонстрация:** открываю `http://localhost:3000/prototype/web-screens/hints` → кнопка «Отчёт» в тулбаре → общий дашборд → меняю период → показываю диаграмму.

**Нюансы (из обсуждения):**

- **Период — только три значения.** Руслан: «Чтобы хитрые люди через пост-запросы период не задали, период заложить через параметры непросто. Три значения.»
- **«Срабатывание» vs «Продаж».** Руслан: «Несрабатывание, продаж — срабатывание путает всех.» Переименовали:
  - «Показов» → «Срабатывание» (сколько раз плагин вывел подсказку)
  - «Срабатываний» → «Продаж» (сколько раз гость купил по подсказке)
- **Не per-hint, а общий.** Руслан: «Нам нужен общий отчёт, где у тебя все подсказки ... Ты должен на этом общем борде показать все подсказки, которые у тебя были в обозначенный период работали.»
- **Круговая диаграмма** — инициатива Руслана: «И какую-нибудь диаграммку залепить.» Мы добавили donut chart

**Это пример того, как встреча с разработчиком меняет прототип.** Спецификация описывала per-hint панель. Руслан сказал — нет, нужен общий дашборд. Мы переделали.

**Статус:** ✅ Прототип переделан. Спецификация требует актуализации (в спеке описан per-hint подход).

## 4. Скоуп 4: Права доступа (3 минуты)

---

### 4.1. Права доступа DS

**Суть:** сейчас зарегистрированы только 2 кода DS (`APP_ARRIVALS`, `APP_DIGITAL_VOICE`), оба на корневом уровне, без папки. Бэкенд не проверяет права DS. Мы:
- Создаём папку «Экраны и звуки» верхнего уровня
- Регистрируем 3 недостающих кода: `APP_CUSTOMER_SCREEN`, `APP_HINTS`, `APP_MENU_BOARD`
- Переносим существующие коды в подпапки
- Добавляем бэкенд-проверку

**Что сделано в прототипе:**
- Отдельный раздел на главной: «Web — панели администрирования» → «Общие настройки»
- Страница «Права доступа» в стиле Web:
  - Левая панель: список ролей + поиск
  - Правая панель: дерево прав с чекбоксами
  - Новая папка «Экраны и звуки» с 5 подпапками
  - 3 новых кода помечены бейджем «Новый»
  - Сохранение состояния в localStorage
  - Поиск по ролям и по названию/коду права

**Демонстрация:** открываю `http://localhost:3000/prototype/web-settings` → показываю дерево → папка «Экраны и звуки» → 5 подпапок → новые коды с бейджем «Новый».

**Нюансы (из обсуждения с Мишей и Русланом):**

- **Подпапки — не нужны.** Миша: «Под папки не надо, а лучше все их отдельно. Просто в одном списке на уровень экрана и звуки поднять.» Руслан подтвердил: «Да, конечно, это на первом уровне. Не надо это папка в папке.» Мы убрали промежуточные подпапки — теперь 5 кодов `APP_*` на первом уровне внутри «Экраны и звуки»
- **Фронтенд-скрытие разделов** при отсутствии права — работает сейчас, без доработок
- **Плагины используют `LicenseModuleId`** — не меняется
- **Миграция ролей — ручная.** После регистрации кодов администратор сам назначит права

**Это чисто бэкенд-задача.** Прототип — витрина целевого UI, чтобы команда видела, как будет выглядеть страница после внедрения.

**Статус:** ✅ Прототип готов (v1.1). Спецификация готова. Ждёт бэкенд-реализации.

### 5.4. Как работать с прототипами

Прототипы доступны локально: `http://localhost:3000`. Все данные сохраняются в localStorage — можно «поиграться» без последствий. Кнопка Reset в top-bar сбрасывает данные к мок-значениям.

**Навигация по прототипам:**
- **Главная:** `http://localhost:3000` — все прототипы сгруппированы по разделам
- **Web-прототипы:** боковое меню слева — «Электронная очередь», «Доска меню», «Подсказки», «Настройки»
- **Права доступа:** `http://localhost:3000/prototype/web-settings`
