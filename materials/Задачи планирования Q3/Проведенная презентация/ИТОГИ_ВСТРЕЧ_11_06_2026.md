## 2. Менюборды (Доска меню)

### 2.1. Общая концепция

Менюборды — **новая сущность**, копия Arrivals (Электронной очереди), со своими особенностями:
- Сохраняется **мультиэкранность** (несколько мониторов)
- Сохраняется **настройка экранов**
- **Исключена** колонка «Компания» из списка кассовых аппаратов (кампании перенесены в элементы Advertise)
- **Исключены терминальные группы** (см. ниже)

### 2.2. Терминальные группы — ИСКЛЮЧЕНЫ

**Вопрос Алисы:** нужны ли терминальные группы в менюбордах?

**Решение Руслана:** ❌ **НЕ НУЖНЫ**

**Обоснование:** в Arrivals терминальные группы нужны для маршрутизации заказов из кухни. В менюбордах данных из кухни нет — там жёстко заданные позиции меню и реклама. Терминальная группа не имеет смысла.

### 2.3. Элемент MenuList

**Важное уточнение (встреча с Ваней):**

MenuList должен базироваться на элементе **«Состав заказа»** (он же чек-лист), который наследуется от объекта «Чек». Это **не полностью новый элемент**, а существующий объект с расширениями:
- Показывает **модификаторы** (уже умеет)
- Нужно добавить: **КБЖУ** и **размерность** (основное отличие от базового объекта)
- Подключается **внешнее меню** (не чек)
- Все остальные свойства — как у исходного объекта (передвижение местами и т.д.)

**Решение:** MenuList и Control — **НЕ объединять** (см. ранее зафиксированное решение в CORRECTIONS_AFTER_MEETING.md). MenuList проще для пользователя (показывает модификаторы «из коробки»), Control — для продвинутой кастомизации.

### 2.4. Элемент Advertise — множественные кампании

**Решение (уже зафиксировано в CORRECTIONS_AFTER_MEETING.md):**
- Снять ограничение на **количество** элементов Advertise (было: 1, стало: сколько угодно)
- Каждому элементу — **мультивыбор** кампаний (было: single-select, стало: multi-select)
- Кампании могут переключаться/чередоваться

### 2.5. Поиск в палитре

**Решение:** ✅ Добавить поиск (как и в задаче «Группировка элементов»)

## 4. Отчёт подсказок — **ПОЛНАЯ ПЕРЕДЕЛКА** 🔴

### Контекст встречи

**Текущий прототип:** кнопка «Отчёт» в каждой строке таблицы подсказок → открывает панель для одной конкретной подсказки с KPI-карточками.

**Решение встречи:** полная переделка в **общий дашборд** по всем подсказкам.

### Ключевые решения

#### А. Общий дашборд (не per-hint)

- ❌ Было: кнопка «Отчёт» в каждой строке таблицы
- ✅ Стало: **одна общая кнопка «Отчёт»** в toolbar страницы подсказок
- Открывает дашборд по **ВСЕМ** подсказкам

#### Б. Метрики — переименование и разделение

**Переименование (Кирилл + Руслан):**
- ❌ «Показов» → ✅ «Срабатывание» (сколько раз плагин вывел подсказку)
- ❌ «Срабатываний» → ✅ «Продаж» (сколько раз гость купил по подсказке)
- ✅ «Конверсия» — остаётся (Продаж / Срабатывание × 100%)

**Разделение кассир/гость (Руслан настоял):**

Добавить **чекбокс «Учитывать показы кассиру»**:

| Чекбокс | Поведение |
|---------|-----------|
| Выключен | Показы только гостю. Колонка «Показы гостю» |
| Включен | Показы и гостю, и кассиру. Две колонки: «Показы гостю» и «Показы кассиру» |

**Логика (от Руслана):**
- Срабатывание = показ гостю + показ кассиру (если включено)
- Продажи — **единые** (факт покупки, без разницы где был показ)
- Конверсия — **общая** (от всех срабатываний)
- Если обе галочки — цифры могут совпадать (одна и та же подсказка показана и там и там)
- Если только гость — в колонке кассира 0

#### В. Donut-график (круговая диаграмма)

- Тип: **donut** (бублик), не линейный
- Показывает **процентное соотношение** срабатываний всех подсказок
- Сегменты = названия подсказок
- Ваня подтвердил: технически реализуемо, данные прилетают → отрисовать можно

#### Г. Фильтр периода — жёстко фиксированный

- **Только три значения:** День, Неделя (по умолчанию), Месяц
- ❌ **ЗАПРЕЩЁН** произвольный период (кастомные даты)
- ❌ **ЗАПРЕЩЁН** выбор через POST-запросы с произвольными параметрами
- Причина (Кирилл): «чтобы хитрые люди через пост-запросы период не задали»

#### Д. Дополнительные фичи (не в изначальном прототипе)

| Фича | Инициатор | Решение |
|------|-----------|---------|
| **Поиск** по подсказкам | Юра / Руслан | ✅ Добавить (но «чуть позже») |
| **Копирование** подсказки | Алиса | ✅ Добавить. Метод на бэке есть (Рома подтвердил) |
| **Тултипы** к метрикам | Паша | ✅ Добавить (чтобы было понятно без документации) |
| **Кнопка «Документация»** | Руслан | 🔵 На будущее (в продуктах Web есть кнопка ссылки на доку) |

#### Е. Техническая реализация

- Данные пишутся в `ExternalData` для заказов (плагин → Front SDK → Web)
- Отчёт собирает метрики по запросу и возвращает для отображения в браузере

## 6. Редактирование контрола из-под темы

### Контекст встречи

Кирилл показал прототип: в инспекторе области кнопка «Редактировать контрол» → переход в редактор контрола → возврат обратно в тему.

### Ключевые решения

#### А. UX-поток

```
Редактор темы → Нажать «Редактировать контрол»
  → [СОХРАНИТЬ ТЕМУ] (обязательно!)
  → Переход в редактор контрола (с query-параметром: ID темы для возврата)
  → Редактировать контрол → Сохранить
  → Нажать «Назад» → Возврат в редактор темы (по сохранённому ID)
```

#### Б. Сохранение темы при выходе

- **Решение:** сохранять тему **автоматически** перед переходом в редактор контрола
- **Причина (Руслан):** «MacOS при свайпе вправо-влево перегружает страницы и ни хрена наш конструктор даже не мычит о том, что он тему не сохраняет»
- **Альтернатива:** задавать вопрос (плашка с подтверждением) — «давайте посмотрим по ходу»

#### В. Сохранение контрола — логика «Save as copy»

**Уже реализовано (Люся делала):**

- Система понимает, где используется контрол (в каких темах)
- Если контрол используется в других темах → **предупреждение**
- Два варианта:
  1. **Сохранить везде** — изменения применятся ко всем темам (после синхронизации экранов)
  2. **Сохранить как копию** — создаётся новый контрол, область в теме перепривязывается к копии

**Решение:** логика уже есть, нужно только «прикрутить кнопочку и пробросить данные туда-сюда» (Ваня). Оценка не меняется.

#### Г. Control-in-control — НЕ делаем

**Вопрос Руслана:** «А в контроле есть еще один контрол. Можем так?»

**Решение:** ❌ **НЕТ**
- Слишком сложное дерево (Рома: «очень тяжелое дерево, перегрузят»)
- Только **один уровень** погружения
- Именно поэтому появился MenuList — чтобы не делать control-in-control для отображения модификаторов

#### Д. Одиночный объект (новый тип области)

**Решение Руслана:** добавить **третий тип области** — «Одиночный объект»:
- Не лист, не динамически заполняемая область
- Жёстко **один** объект (контрол) заданного размера
- Использование: 10 таких объектов = 10 бургеров на менюборде
- Подключение к внешнему меню (цены и картинки — из внешнего меню)
