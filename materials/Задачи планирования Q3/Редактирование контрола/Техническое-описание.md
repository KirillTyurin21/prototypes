# Техническое описание: Редактирование контрола из темы

> **Дата:** 2026-06-11
> **Задача:** Скоуп-1-Конструктор / Задача-3-Редактирование-контрола
> **Прототип:** Web Screens (Электронная очередь + Доска меню)

---

## 1. Что сделано

Раньше: чтобы отредактировать контрол, администратор уходил из редактора темы → шёл в справочник контролов → искал нужный → редактировал → возвращался в тему. Это разрывало workflow.

Теперь: в редакторе темы, прямо в инспекторе области (правая панель), появилась кнопка **«Редактировать контрол»**. Нажал → перешёл в редактор контрола → отредактировал → «Назад» → снова в теме. Всё в рамках одного интерфейса.

## 2. Техническая реализация

### 2.1. Что поменяли

| Файл | Что изменилось |
|------|---------------|
| `area-element-inspector.component.ts` | Добавлен `@Output() editControl` и кнопка «Редактировать контрол» под выпадающим списком выбора контрола |
| `arrivals-theme-editor-screen.component.ts` | Подписка на `(editControl)` → автосохранение темы → переход в редактор контрола с `returnUrl` |
| `menuboard-theme-editor-screen.component.ts` | Аналогично для Доски меню |
| `arrivals-control-editor-screen.component.ts` | Кнопка «Назад» научилась возвращать в редактор темы (а не только в справочник) |

### 2.2. Как работает навигация

```
Редактор темы
  │
  ├─ Клик на область с контролом → инспектор области (правая панель)
  │
  ├─ Кнопка «Редактировать контрол» (видна только если areaControlId не пустой)
  │
  ├─ Автосохранение темы в localStorage
  │
  ├─ Роутинг: /arrivals-control-editor/:id?return=theme-editor&themeId=123
  │
  ▼
Редактор контрола (существующий экран)
  │
  ├─ Canvas + элементы + инспектор (всё работает как раньше)
  │
  ├─ Кнопка «Назад» → проверяет query-параметр return:
  │   ├─ return=theme-editor → /arrivals-theme-editor/:themeId
  │   ├─ return=menuboard-theme-editor → /menuboard-theme-editor/:themeId
  │   └─ без return → /arrivals-controls (старое поведение)
  │
  ▼
Снова редактор темы (тема загружается из localStorage)
```

### 2.3. Ключевые детали

**Автосохранение перед уходом.** Перед переходом в редактор контрола вызывается `this.save()` — тема сохраняется в localStorage. Это гарантирует, что даже несохранённые правки не потеряются.

**Реальный themeId.** Для новой темы (`/new`) передаётся не строка `'new'`, а реальный числовой ID (`this.theme.id = Date.now()`). Иначе при возврате компонент не найдёт сохранённую тему.

**Не создавали новых файлов.** Кнопка встроена в существующий `area-element-inspector`. Редактирование — в существующем `arrivals-control-editor-screen`. Ни одного нового компонента.

## 3. UX flow (пользовательский сценарий)

1. Администратор открывает редактор темы (Электронная очередь / Доска меню)
2. Кликает на область, к которой прикреплён контрол
3. В правой панели инспектора, под выпадающим списком «Контрол», видит кнопку **«Редактировать контрол»** (с иконкой карандаша)
4. Нажимает → попадает в полноценный редактор контрола
5. Редактирует элементы: двигает, меняет текст, шрифты, границы, цвета
6. Жмёт «Сохранить» → «Назад»
7. Возвращается в редактор темы — изменения контрола уже отображаются на canvas

## 5. Почему так, а не модальное окно

Изначально в спецификации обсуждался вариант с модальным окном (двойной щелчок → модалка с элементами контрола). Но:

- Пришлось бы дублировать весь инспектор элементов (уже есть в редакторе контрола)
- Пришлось бы решать конфликт dblclick с drag-and-drop на canvas
- Пришлось бы создавать новый компонент и новые стили

**Выбрали путь с навигацией** — переиспользовали существующий полнофункциональный редактор контрола. Меньше кода, меньше багов, привычный интерфейс для пользователя.
