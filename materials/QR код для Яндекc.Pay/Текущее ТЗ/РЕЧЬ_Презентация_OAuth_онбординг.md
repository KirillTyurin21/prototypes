## Часть 1. Зачем мы здесь

Ребят, давайте начнём с верхнего уровня.

Мы дорабатываем панель Яндекс.Пэй в iikoWeb — автоматизируем подключение ресторанов через OAuth 2.0.

Проблема простая. Сейчас, чтобы подключить ресторан к Яндекс.Пэй, ресторатор должен руками: зайти в личный кабинет Яндекса, найти и скопировать User Token, вставить его в настройках iikoWeb, потом ещё привязать QR-табличку. Это занимает часы, порождает ошибки копипаста — и каждый раз нужно повторять для каждого ресторана.

Мы это автоматизируем. После доработки процесс будет такой: нажал «Войти с Яндекс ID», заполнил реквизиты, подал заявку — и всё. User Token приходит автоматически, QR-таблички привязываются сами.

Давайте я пройдусь по всей архитектуре решения — от бизнес-логики до таблиц в базе.

## Часть 2. Что трогаем, а что нет

Важный момент — мы не трогаем всё подряд.

**Трогаем:**
- **iikoWeb** — основной фронт. Там появится блок OAuth-авторизации, формы создания организации и мерчанта, экран статусов заявок.
- **iikoWeb Backend** — серверная часть. Здесь будут храниться OAuth-токены, проксироваться запросы к API Яндекса, и дополнительная логика автосохранения.
- **Плагин iikoFront** — минимально. Вообще, он не узнает, что что-то поменялось — он по-прежнему дёргает тот же endpoint и получает тот же токен.

**НЕ трогаем:**
- Платёжное API — операции оплаты, возврата, отмены — всё работает как раньше.
- iikoOffice — не затрагивается.
- Мобильные приложения — не затрагиваются.

## Часть 3. Иерархия сущностей — «кто есть кто»

Прежде чем нырять в детали, давайте разберёмся с иерархией сущностей на стороне Яндекс.Пэй. Это важно для понимания всей последующей логики.

Иерархия такая — четыре уровня:

**Пользователь** (Яндекс ID — аккаунт, через который авторизуемся) → **Partner** (организация, юрлицо — ООО или ИП) → **Merchant** (торговая точка, конкретный ресторан) → **POS** (QR-табличка, привязывается к кассе iiko).

Теперь по каждой сущности подробнее.

### Partner — это юрлицо

Partner идентифицируется по ИНН. Создаётся через `POST /partners`. Ключевой момент — **ИНН является ключом идемпотентности**: если организация с таким ИНН уже есть, API не создаст дубликат, а вернёт существующую запись. Это штатное поведение, не ошибка.

Какие данные передаём при создании партнёра? Из 16 полей **обязательное только одно** — `registration_data.tax_ref_number`, это и есть ИНН. Всё остальное опционально, потому что Яндекс умеет подтягивать данные по ИНН через DaData.

Перечислю все поля, чтобы было понимание объёма формы:

- `name` — название организации.
- Блок `registration_data`: `tax_ref_number` (ИНН, 10 цифр для юрлица или 12 для ИП — **единственное обязательное поле**), `ogrn` (13 или 15 цифр), `kpp` (9 цифр, только для юрлиц), `legal_address`, `postal_address`, `postal_code` (6 цифр), `full_company_name`, `ceo_name`, `url`.
- Блок `contact`: `email`, `phone` (формат +7XXXXXXXXXX), `last_name`, `first_name`, `middle_name`.

Итого 15 полей в запросе, плюс `partner_id` генерируется Яндексом в ответе.

### Merchant — это конкретный ресторан

Мерчант привязан к Partner. Создаётся через `POST /merchants/register`. Одна заявка = один ресторан, это MVP-ограничение.

Здесь полей больше, и большинство **обязательные**. Перечислю:

- `partner_id` — откуда берётся? Из предыдущего шага, когда мы создали или выбрали организацию.
- `merchant.name` — название торговой точки. Предзаполняется из дерева ресторанов iiko.
- `merchant.address` — физический адрес. Тоже предзаполняется из iiko, уровень RMS.
- `onboarding_data.mcc` — MCC-код, категория бизнеса. Строка из 4 цифр. По умолчанию ставим `"5812"` — «Рестораны». Список всех кодов получаем через `GET /mcc-codes`.
- Банковские реквизиты — три обязательных поля: `bank_details.bik` (ровно 9 цифр), `bank_details.settlement_account` (ровно 20 цифр), `bank_details.correspondent_account` (ровно 20 цифр).
- Контактные данные заявки — `communication_contact.email`, `phone`, `last_name`, `first_name` — все обязательные. `middle_name` — опционально.
- `poses_count` — количество QR-табличек (касс). Опционально. Берётся из количества выбранных чекбоксов в интерфейсе, максимум 10 на заявку.

Все обязательные — это 11 полей из 13. Необязательных только два: `middle_name` и `poses_count`.

### Статусы мерчанта

После подачи заявки мерчант может находиться в одном из трёх состояний (поле `registration_status` в ответах API):

- `"processing"` — заявка на рассмотрении у Яндекса, ориентировочно 2 дня. Мы поллим статус не чаще раза в 10 минут.
- `"active"` — одобрено, мерчант активен. Мы автоматически запрашиваем User Token.
- `"failed"` — отклонено. Причину Яндекс **не возвращает** — нет поля `reason`. Показываем «Отклонено», рекомендуем обратиться в поддержку Яндекс.Пэй.

### POS — QR-табличка

Последний уровень. POS привязывается к терминалу iikoFront. У каждого POS есть `pos_id`. Ограничение — максимум 10 QR-табличек на один Merchant ID. Если касс больше 10 — подаётся вторая заявка, создаётся новый `merchant_id`, каждый со своим набором QR-кодов.

## Часть 4. Два контура авторизации — не путаем

Это ключевой архитектурный момент. В системе работают **два полностью независимых контура авторизации**, и их важно не смешивать.

**Первый контур — OAuth 2.0, новый.** Используется для управления организациями и мерчантами через Vendor Cabinet API. Авторизация через Bearer-токен в заголовке `Authorization: Bearer <access_token>`. Base URL продакшена: `https://console.pay.yandex.net/api/vendor/v1`, тестовый: `https://test.console.pay.yandex.net/api/vendor/v1`. Токен хранится в серверной сессии iikoWeb Backend.

**Второй контур — Api-Key + User-Token, существующий.** Используется плагином iikoFront для платёжных операций — оплата, возврат, отмена. Совершенно другие заголовки: `Api-Key: <key>` и `User-Token: <token>`. Совершенно другой URL: `https://api.qr.yandex.net/api/v1/s1` (прод) и `https://tester.newpay.pro/api/v1/s1` (тест).

Почему это важно? Если OAuth упадёт — оплата на кассах **продолжит работать**. Контуры изолированы. Мы не трогаем второй контур вообще.

## Часть 5. Как работает OAuth-авторизация

### Регистрация приложения

Первый шаг — мы однократно регистрируем OAuth-приложение на https://oauth.yandex.ru/. Получаем `client_id` (публичный, можно передавать на фронтенд) и `client_secret` (секретный, **только на backend**, никогда на фронтенд, никогда в логи). Указываем `redirect_uri` — это `https://{iiko-web-domain}/api/oauth/callback`, один на весь SaaS. Тип токена рекомендуется **продлеваемый** (TTL не менее 1 года).

Тут есть открытый вопрос: конкретные scopes для Vendor Cabinet API пока не раскрыты в документации Яндекса. Мы уточним при регистрации или у менеджера. Если scope указан неверно — API вернёт HTTP 403 «Недостаточно прав».

### Authorization Code Flow — пошагово

Используем стандартный Authorization Code Flow. Давайте пройдём по шагам.

**Шаг 1.** Пользователь нажимает «Войти с Яндекс ID» в iikoWeb.

**Шаг 2.** Фронтенд перенаправляет браузер на `GET https://oauth.yandex.ru/authorize`. В URL передаём четыре параметра:
- `response_type=code` — фиксированное значение, мы хотим авторизационный код.
- `client_id` — ID нашего приложения из регистрации.
- `redirect_uri` — куда Яндекс вернёт пользователя. Должен **точно** совпадать с зарегистрированным.
- `state` — случайная криптографически стойкая строка для CSRF-защиты. Backend генерирует её (UUID v4 или `crypto.randomBytes`), сохраняет в сессии, а при callback проверяет совпадение.

**Шаг 3.** Яндекс показывает форму логина. Пользователь вводит логин/пароль, подтверждает доступ.

**Шаг 4.** Яндекс перенаправляет браузер обратно на наш `redirect_uri` с параметрами `code` (авторизационный код, живёт ~30 секунд) и `state`.

**Шаг 5.** Backend принимает callback. Первое — проверяет `state`: сравнивает с сохранённым в сессии. Если не совпадает — отклоняет запрос и логирует `[SECURITY INCIDENT]`. Если совпадает — делает server-to-server запрос `POST https://oauth.yandex.ru/token`. В теле:
- `grant_type=authorization_code`
- `code` — тот самый авторизационный код
- `client_id`
- `client_secret` — вот тут он нужен, **только** в этом server-to-server вызове

**Шаг 6.** Яндекс отвечает JSON-ом с четырьмя полями:
- `access_token` — Bearer-токен для Vendor Cabinet API. Его мы будем подставлять в заголовок `Authorization: Bearer <token>` при каждом вызове.
- `refresh_token` — токен для обновления `access_token` без повторной авторизации пользователя.
- `token_type` — всегда `"bearer"`.
- `expires_in` — время жизни в секундах, обычно 31536000 (1 год для продлеваемых токенов).

Оба токена сохраняются в серверной сессии. **Не в БД** — при завершении сессии токены удаляются, нужна повторная авторизация.

### Обновление токена

Когда `access_token` истечёт или API вернёт 401, backend пытается обновить его. Тот же `POST https://oauth.yandex.ru/token`, только `grant_type=refresh_token` и вместо `code` передаём `refresh_token`. Ответ идентичный — новая пара `access_token` + `refresh_token`.

Если и `refresh_token` невалиден (ответ `invalid_grant`) — очищаем оба токена из сессии и предлагаем пользователю повторный вход.

Рекомендация Яндекса — обновлять долгоживущие токены раз в 3 месяца. Пользователь также может отозвать доступ через https://id.yandex.ru/ или сменой пароля — тогда наш токен просто перестанет работать, и мы обработаем это стандартно через 401.

### Кнопка «Выйти»

Когда пользователь нажимает «Выйти» — backend вызывает `POST /api/oauth/logout`, очищает `access_token` и `refresh_token` из серверной сессии. UI переключается в состояние «Не авторизован».

Критически важно: «Выйти» очищает **только** Bearer-токен. Ранее полученные User Token и привязки QR-табличек **продолжают работать** на кассах. Сессия Яндекс ID в браузере тоже сохраняется — повторное «Войти» может пройти без пароля.

### Обработка ошибок OAuth

Основные кейсы:
- Пользователь отменил авторизацию — `error=access_denied` в callback. Закрываем popup, показываем кнопку «Войти».
- Невалидный `state` — CSRF-атака. Отклоняем callback, логируем `[SECURITY INCIDENT]`.
- Авторизационный код истёк (~30 секунд) — `invalid_grant`. Перенаправляем на повторную авторизацию.
- Popup заблокирован браузером — `window.open` вернул `null`. Показываем подсказку «Разрешите всплывающие окна».
- Таймаут callback — 5 минут. Закрываем popup по таймеру.

## Часть 6. Vendor Cabinet API — что за зверь

Это новый API от Яндекса, версии v1. Content-Type — JSON. Авторизация — Bearer-токен из OAuth. Формат ответа стандартный: `{ "code": HTTP-код, "data": полезная нагрузка, "status": "success" }`.

Всего 9 методов, сгруппированных по сущностям. Давайте пройдёмся по каждому.

### Метод 1: `GET /partners` — получить организации

Просто GET-запрос, без тела. Возвращает массив `partners[]` — все организации текущего Яндекс-аккаунта. Каждый элемент содержит `partner_id`, `name`, блок `registration_data` (ИНН, ОГРН, КПП...) и блок `contact` (email, телефон, ФИО). Массив может быть пустым, если организаций нет.

### Метод 2: `POST /partners` — создать организацию

Тело запроса — те самые 15 полей, которые я описывал в Части 3: блоки `name`, `registration_data` и `contact`. Обязательный только `tax_ref_number`.

В ответе получаем `partner_id` — это ID партнёра, который нам понадобится для всех следующих операций.

Ключевая фишка — идемпотентность по ИНН. Если ИНН уже зарегистрирован, API вернёт 200 OK с существующим партнёром, без создания дубликата. Коды ошибок: 400 — невалидный ИНН, 401 — токен невалиден, 403 — нет нужного scope.

### Метод 3: `POST /merchants/register` — подать заявку на мерчанта

Самый «толстый» запрос. Тело содержит `partner_id` (из предыдущего шага) и массив `merchants[]`. В MVP массив содержит **ровно 1 элемент** — одна заявка = один ресторан.

Каждый элемент — это вложенная структура: `merchant` (name, address), `onboarding_data` (mcc), `bank_details` (bik, settlement_account, correspondent_account), `communication_contact` (email, phone, last_name, first_name, middle_name) и `poses_count`.

В ответе получаем `merchant_id` и `registration_status: "processing"`. Массив `poses[]` при создании пустой — QR-таблички появятся после одобрения.

### Метод 4: `GET /merchants/{merchant_id}/status` — поллинг статуса

В URL передаём `merchant_id` из предыдущего ответа. Возвращает `registration_status` — `"processing"`, `"active"` или `"failed"`. Если `"active"` — в ответе также будет массив `poses[]` с `pos_id` каждой QR-таблички.

Обратите внимание: **нет поля `reason` или `failure_reason`**. Если отклонили — мы не знаем почему, просто показываем «Отклонено».

Поллинг: не чаще 1 раза в 10 минут, плюс при каждом открытии страницы. Ожидаемое время обработки Яндексом — около 2 дней. Прекращаем поллинг, когда статус ≠ `"processing"`.

### Метод 5: `GET /merchants?partner_id={id}` — список мерчантов

GET-запрос с query-параметром `partner_id`. Возвращает массив мерчантов партнёра — `merchant_id`, `partner_id`, `registration_status`, `poses[]`. Нужен для загрузки существующих мерчантов при повторной авторизации.

### Методы 6, 7, 8 — управление User Token

**Метод 6: `GET /merchants/{merchant_id}/usertoken`** — список токенов мерчанта. Возвращает массив `tokens[]`, у каждого: `token_id`, `token_value` (это и есть User Token для платёжного API — то значение, которое плагин iikoFront подставляет в заголовок `User-Token`), `created_at` и `status` (`"active"`, `"inactive"`, `"revoked"`). У одного мерчанта может быть до 3 валидных токенов одновременно.

**Метод 7: `POST /merchants/{merchant_id}/usertoken`** — создать новый токен. Тело запроса **пустое** — просто `{}`. В ответе — `token_id` и `token_value`. Важно: при создании нового токена старый **не удаляется** автоматически. Если уже есть 3 токена — вернётся 409 Conflict.

Это ключевой метод, потому что именно здесь мы получаем User Token, который потом автоматически сохраним в БД iiko (подробнее — в Части 10, Endpoint K).

**Метод 8: `DELETE /merchants/{merchant_id}/usertoken/{token_id}`** — удалить токен. Ответ — 200 или 204.

### Метод 9: `GET /mcc-codes` — справочник категорий

Возвращает массив `mcc_codes[]`, у каждого: `mcc` (4-значная строка, например `"5812"`), `name` (например «Рестораны»), `description`. Мы показываем это в dropdown: «5812 — Рестораны».

Кешируем этот справочник на backend in-memory, TTL 24 часа. Если API недоступен — используем stale-кеш. Если кеш пуст и API лежит — ошибка.

## Часть 7. Happy Path — полный путь пользователя

Теперь давайте пройдём весь путь пользователя от начала до конца. Я буду указывать, где автоматические шаги, а где действия пользователя.

**Шаг 1 — пользователь.** Открывает раздел «Интеграции» → «Яндекс.Пэй» в iikoWeb. Видит блок авторизации с кнопкой «Войти с Яндекс ID».

**Шаг 2 — пользователь.** Нажимает кнопку. Frontend вызывает наш `GET /api/oauth/authorize`, backend генерирует `state`, формирует URL, открывает popup.

**Шаг 3 — пользователь.** Вводит логин/пароль на Яндекс ID, подтверждает доступ.

**Шаг 4 — автоматически.** Яндекс возвращает `code` и `state` на наш callback. Backend проверяет `state`, обменивает `code` на `access_token` и `refresh_token` через server-to-server `POST /token`.

**Шаг 5 — автоматически.** Сразу после авторизации backend вызывает `GET /partners`. Если организаций нет — кнопка «Создать организацию». Если одна — выбирается автоматически. Если несколько — список для выбора.

**Шаг 6 — пользователь.** Нажимает «Создать организацию», открывается форма.

**Шаг 7 — пользователь.** Заполняет ИНН (минимум) + остальные поля по желанию. Нажимает «Создать» → backend вызывает `POST /partners`. Ответ мгновенный — получаем `partner_id`. Если ИНН уже есть — возвращается существующий партнёр.

**Шаг 8 — автоматически.** Backend вызывает `GET /merchants?partner_id={id}`. Для новой организации список пуст → кнопка «Подать заявку на подключение».

**Шаг 9 — пользователь.** Выбирает ресторан из дерева iiko. Система предзаполняет название и адрес.

**Шаг 10 — пользователь.** Заполняет MCC-код (dropdown, по умолчанию «5812 — Рестораны»), банковские реквизиты (БИК, расчётный счёт, корр. счёт), контакт (email, телефон, ФИО). Выбирает кассы чекбоксами (максимум 10). Нажимает «Подать заявку» → backend вызывает `POST /merchants/register` → получаем `merchant_id` со статусом `"processing"` → Toast «Заявка подана».

**Шаг 11 — автоматически.** Backend запускает поллинг `GET /merchants/{merchant_id}/status` — не чаще раза в 10 минут. Ожидание — около 2 дней.

**Шаг 12 — автоматически.** Поллинг возвращает `registration_status: "active"` → статус в UI меняется на «Подключен» (зелёный) → поллинг прекращается.

**Шаг 13 — автоматически.** Backend вызывает `POST /merchants/{merchant_id}/usertoken` → получает `token_value`.

**Шаг 14 — автоматически.** Это ключевой момент. Backend **программно** вызывает наш существующий endpoint `POST /api/yandex-pay/codes` и передаёт ему `storeId` (ID ресторана из маппинга), `code` равный полученному `token_value`, `source: "oauth"` и `merchant_id`. Таким образом User Token попадает в БД iiko автоматически. Поле «Ключ» в UI заполняется само и становится readonly с подписью «Получен автоматически».

**Шаг 15 — автоматически.** Загрузка QR-табличек через существующие endpoints `POST /api/yandex-pay/codes/{codeId}/refresh` и `GET /api/yandex-pay/accounts/{storeId}`.

**Шаг 16 — автоматически + пользователь.** QR-коды от Яндекса обезличены (не привязаны к конкретным кассам), поэтому iiko автоназначает их на кассы в произвольном порядке. После этого пользователь может переназначить вручную через dropdown. Привязка сохраняется мгновенно (без кнопки Save) через `POST /api/yandex-pay/terminal-accounts`.

**Шаг 17 — пользователь.** Выбирает табличку по умолчанию через `POST /api/yandex-pay/codes/{storeId}/default-account` → ресторан полностью подключён, оплата готова к работе.

Итого: **из 17 шагов пользователь активно участвует в 7-8**, остальное делает система.

## Часть 8. Другие сценарии — что может пойти не так

Кроме Happy Path, давайте пройдёмся по альтернативным сценариям.

**Организация уже существует.** Пользователь вводит ИНН, а она уже зарегистрирована. `POST /partners` вернёт 200 OK с существующим партнёром — дубликат не создаётся. Показываем: «Организация с ИНН такой-то уже зарегистрирована. Данные подтянуты автоматически». И загружаем мерчантов через `GET /merchants`.

**Повторная авторизация — всё уже подключено.** Загружаются организации и мерчанты, все со статусом `"active"` показываются зелёным. Ранее настроенные интеграции продолжают работать.

**Чужой аккаунт.** `GET /partners` вернёт пустой список или чужие организации. Подсказка: «Нужной организации нет? Нажмите "Выйти" и войдите под корректным аккаунтом».

**Заявка отклонена.** `registration_status: "failed"`. Красная метка «Отклонено», поллинг прекращён. API не возвращает причину — рекомендуем обратиться в поддержку Яндекс.Пэй. Кнопка «Подать повторно» → создаётся **новый** `merchant_id`.

**Больше 10 касс.** Лимит API — 10 POS на один Merchant ID. Если касс 15 — подаём две заявки: первая на 10 касс, вторая на 5. Два `merchant_id` для одного ресторана, каждый со своим набором QR-кодов.

**Смена OAuth-аккаунта.** «Выйти» очищает **только** Bearer-токен. User Token в БД и привязки QR-табличек — всё продолжает работать. Оплата на кассах не пострадает.

## Часть 9. Изменения в интерфейсе

Теперь давайте поговорим про UI. Что конкретно увидит пользователь.

### Блок авторизации

Размещается в DetailPanel, **выше** поля «Ключ». Два состояния:

**Не авторизован:** заголовок «Автоматическое подключение», пояснительный текст, кнопка «Войти с Яндекс ID» по гайдлайнам Яндекса (через YaAuthSuggest SDK), и горизонтальный разделитель «или» — ниже которого остаётся ручной ввод.

**Авторизован:** индикатор «Авторизован: user@yandex.ru» зелёным, кнопка «Выйти» красным, селектор организации (dropdown из `GET /partners`), и список мерчантов с их статусами.

Важно: ручной ввод ключа **остаётся доступным** ниже разделителя «или». Это альтернативный способ подключения. Если ключ получен через OAuth — поле становится readonly с подписью «Получен автоматически через Яндекс ID».

### Форма организации

Модальное окно, 15 полей. Из них обязательное только ИНН — кнопка «Создать» заблокирована до заполнения валидного ИНН. Каждое поле маппится один-в-один на поле в `POST /partners`, я описал соответствие в Части 3.

### Форма заявки на мерчанта

Тут интереснее. Сверху — dropdown выбора ресторана из дерева iiko, который предзаполняет название и адрес. Далее — MCC-код (dropdown из справочника `GET /mcc-codes`, по умолчанию «5812 — Рестораны»). Банковские реквизиты — три поля в одну строку: БИК (9 цифр), расчётный счёт (20 цифр), корр. счёт (20 цифр). Контактные данные — email, телефон, ФИО.

Внизу — чекбоксы выбора касс. Максимум 10 на заявку — при достижении лимита остальные чекбоксы disabled с подсказкой «Достигнут лимит 10 касс». Уже подключённые кассы тоже disabled — «Подключена (заявка от ДД.ММ.ГГГГ)». Минимум 1 касса — при нулевом выборе кнопка «Подать заявку» неактивна.

Количество выбранных чекбоксов уходит в API как `poses_count`. ID касс запоминаются локально в iiko — Яндексу они **не передаются**.

Если ранее подавалась заявка — контактные данные и реквизиты предзаполняются из последней заявки.

### Экран статусов

Под блоком авторизации. Каждый мерчант — строка с названием, статусом (жёлтый «Обрабатывается» / зелёный «Подключен» / красный «Отклонён»), датой подачи и кнопками действий.

### Секция QR-табличек

При OAuth — поле «Ключ» автозаполняется и становится readonly. Кнопка «Сохранить» скрыта. QR-таблички загружаются автоматически. При OAuth — автоназначение на кассы + возможность ручного переназначения.

## Часть 10. Backend — 13 новых endpoints

Переходим к серверной части. Backend играет роль **прокси-слоя**: фронтенд никогда не обращается к Vendor Cabinet API напрямую. Все запросы идут через наш backend, который подставляет Bearer-токен из серверной сессии.

### Группа OAuth — 4 endpoints

**Endpoint A: `GET /api/oauth/authorize`** — фронтенд вызывает для инициации OAuth. Backend генерирует `state`, формирует URL авторизации, возвращает его.

**Endpoint B: `POST /api/oauth/callback`** — обработка callback от Яндекса. Проверяет `state`, обменивает `code` на токены, сохраняет в сессии.

**Endpoint C: `POST /api/oauth/logout`** — очистка токенов из сессии.

**Endpoint D: `GET /api/oauth/status`** — проверка: авторизован ли пользователь, возвращает `{ authenticated: bool, email: string }`.

### Группа Proxy — 9 endpoints

Endpoints E через M — по одному на каждый метод Vendor Cabinet API. Каждый проксирует запрос, подставляя Bearer-токен.

Особый интерес представляет **Endpoint K: `POST /api/vendor/merchants/{id}/usertoken`**. Он не просто проксирует. У него дополнительная логика:

1. Проксирует `POST /merchants/{id}/usertoken` к Vendor Cabinet → получает `token_value`.
2. Определяет `storeId` через маппинг: ищет в таблице `yandex_pay_merchant_mapping` запись по `merchant_id` → берёт оттуда `store_id`.
3. **Программно** вызывает наш существующий `POST /api/yandex-pay/codes` с телом: `storeId` из маппинга, `code` равный полученному `token_value`, `source: "oauth"` и `merchant_id`.
4. Таким образом User Token автоматически попадает в БД iiko без участия пользователя.

### Кеширование

Кешируем только `GET /mcc-codes` — in-memory, TTL 24 часа. Всё остальное без кеширования — данные могут измениться в ЛК Яндекс.

### Защита от SSRF

Backend проксирует запросы **только** к whitelisted хостам: `oauth.yandex.ru`, `console.pay.yandex.net`, `test.console.pay.yandex.net`. URL формируется на backend из конфигурации, не из данных фронтенда.

### Обработка 401

При получении 401 от Vendor Cabinet API — пытаемся refresh. Если refresh тоже не прошёл (`invalid_grant`) — очищаем оба токена, возвращаем фронтенду «не авторизован», предлагаем повторный вход.

### Логирование

Логируем: факт авторизации, HTTP-статусы, коды ошибок, `account_id`, изменения статусов, CSRF-ошибки.

**Не** логируем: `client_secret`, `access_token`, `refresh_token`, `authorization_code`, `state`, `token_value`, персональные данные в открытом виде.

## Часть 11. Изменения в базе данных

Тут три действия: две новые таблицы и два новых поля в существующей.

### Таблица 1: `yandex_pay_partner_mapping`

Связь «юрлицо iiko» ↔ «партнёр Яндекс.Пэй». Поля:
- `id` — UUID, PK.
- `account_id` — INT, для изоляции данных между аккаунтами iikoWeb. Из текущей сессии.
- `jur_person_id` — UUID, ID юрлица в iiko. Из дерева организаций.
- `partner_id` — VARCHAR(255), ID партнёра в Яндекс.Пэй. Из ответа `POST /partners`.
- `inn` — VARCHAR(12), ИНН. Из формы.
- `partner_name` — VARCHAR(500), название. Из ответа API.
- `created_at`, `updated_at` — TIMESTAMP, автогенерация.

Ограничения: `UNIQUE (account_id, jur_person_id)` — одно юрлицо = один партнёр в рамках аккаунта. Индексы на `account_id` и `partner_id`.

### Таблица 2: `yandex_pay_merchant_mapping`

Связь «ресторан iiko» ↔ «мерчант Яндекс.Пэй». Поля:
- `id` — UUID, PK.
- `account_id` — INT.
- `store_id` — INT, ID ресторана в iiko. Из формы заявки.
- `merchant_id` — VARCHAR(255), из ответа `POST /merchants/register`.
- `partner_id` — VARCHAR(255), ссылка на партнёра.
- `registration_status` — VARCHAR(50): `"processing"`, `"active"`, `"failed"`. Обновляется при каждом поллинге.
- `status_checked_at` — TIMESTAMP, когда последний раз проверяли.
- `created_at`, `updated_at`.

Ограничения: `UNIQUE (account_id, store_id)`. Индексы на `account_id`, `merchant_id`, `partner_id`.

### Изменения в `yandex_pay_code`

Добавляем два поля:
- `source` — VARCHAR(20), NOT NULL, DEFAULT `"manual"`. Принимает значения `"manual"` или `"oauth"`. Default `"manual"` гарантирует обратную совместимость — все существующие записи автоматически получат `source = "manual"` без миграции данных.
- `merchant_id` — VARCHAR(255), NULL. Заполняется только при `source = "oauth"`.

Все три миграции идемпотентные — `CREATE TABLE IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`.

Связи между таблицами — на уровне приложения, не через DDL FOREIGN KEY. Это упрощает миграцию.

Все таблицы содержат `account_id` — все запросы к БД **всегда** фильтруются по `account_id` текущей сессии. Попытка доступа к чужим данным → HTTP 403.

## Часть 12. Обратная совместимость

Это критически важная часть — мы **ничего не ломаем**.

Три принципа:
- **Аддитивность** — OAuth добавляет новое, не удаляет старое.
- **Параллельность** — оба режима (ручной и OAuth) работают одновременно. Разные рестораны могут использовать разные способы.
- **Прозрачность для плагина** — плагин iikoFront работает одинаково. Pull-запрос `POST /api/internal/yandex-pay/code` возвращает токен из БД — ему всё равно, `source = "manual"` или `"oauth"`.

Что **не нужно** делать существующим клиентам: переустанавливать плагин, перенастраивать iikoOffice, удалять старый User Token, менять Api-Key.

### Feature Toggle

OAuth управляется параметром `YandexPayOAuthEnabled`.

Если `false` (по умолчанию) — блок OAuth скрыт, ручной ввод доступен, endpoints `/api/oauth/*` и `/api/vendor/*` возвращают 404. Плагин работает.

Если `true` — блок OAuth виден, ручной ввод тоже доступен, endpoints активны. Плагин работает.

В любом случае — **плагин работает**. Это главное.

### Rollback

Два уровня. **Быстрый** — toggle off, минуты. OAuth скрыт, всё остальное работает.

**Полный** — откат версии iikoWeb + обратные миграции (`DROP TABLE`, `DROP COLUMN`). Все User Token продолжают работать, данные маппинга утрачены — потребуется повторный онбординг через OAuth.

## Часть 13. Ограничения MVP

Зафиксирую основные, чтобы все понимали границы:

1. **Одна заявка = один ресторан.** Batch не делаем.
2. **Максимум 10 касс** на один Merchant ID — ограничение API Яндекса.
3. **Один OAuth-аккаунт на сессию.** Для разных организаций — выход и повторный вход.
4. **Нет фонового обновления статусов.** Обновление при открытии страницы или поллинг не чаще 1 раз в 10 мин.
5. **QR-маппинг только на стороне iiko.** Яндекс не хранит привязку «табличка ↔ касса».
6. **Добавление новых QR-табличек — только через ЛК Яндекс.** API не имеет метода создания POS.
7. **Api-Key не редактируется через OAuth.** Настраивается в XML-конфиге плагина.
8. **OAuth-токены в серверной сессии.** При завершении сессии — повторная авторизация.
9. **Частичные возвраты не поддерживаются.** Ограничение платёжного API Яндекс.

## Часть 14. Обработка ошибок

Основные сценарии ошибок и как мы их обрабатываем.

**Ошибки OAuth** я уже описал в Части 5 — отмена, CSRF, таймаут, popup-блокировка. Все обработаны.

**Ошибки API Яндекса:**
- **400 Bad Request** — невалидные данные. Подсвечиваем невалидные поля: «ИНН должен содержать 10 или 12 цифр», «БИК — 9 цифр». Форма остаётся открытой.
- **401 Unauthorized** — токен истёк или отозван. Пытаемся refresh. При неудаче — сброс авторизации: «Сессия истекла. Необходимо повторно войти через Яндекс ID».
- **403 Forbidden** — нет доступа (чужой ресурс или недостаточно scope). «Нет доступа. Убедитесь, что авторизовались под корректным аккаунтом».
- **404 Not Found** — ресурс удалён в ЛК Яндекс. Обновляем маппинг в БД.
- **500 Internal Server Error** — Яндекс лежит. Toast: «Сервис Яндекс.Пэй временно недоступен. Попробуйте позже». **Не** повторяем автоматически.

**Ошибка получения User Token** — при переходе в `"active"` backend автоматически вызывает `POST /merchants/{id}/usertoken`. Если упал — retry до 3 попыток с интервалом 10 секунд. При неудаче: «Не удалось автоматически получить токен. Попробуйте обновить страницу».

## Часть 15. Плагин iikoFront

Тут коротко. Плагин получает User Token из БД iiko через pull-запрос `POST /api/internal/yandex-pay/code`. Этот endpoint **не меняется**. Плагину всё равно, как токен был получен — вручную или через OAuth.

**Не меняется:** конфигурационный XML, Api-Key, платёжное API, версия плагина. Обновление плагина **не требуется**.

## Часть 17. Открытые вопросы

Перед тем как закончить — есть 11 открытых вопросов, которые нужно закрыть в процессе разработки.

**Q1.** Конкретные **scopes** OAuth-приложения для Vendor Cabinet API — в документации Яндекса не раскрыты. При отсутствии scope — HTTP 403. Уточняем при регистрации. *Адресат: Яндекс.*

**Q2.** Подтверждение **готовности API** к 01.04.2026. Косвенно подтверждено (Q2 2026). *Адресат: Яндекс.*

**Q3.** Точные **имена endpoints**: `POST /merchants` vs `/merchants/register`? `POST /merchants/{id}/tokens` vs `/usertoken`? Ожидается YAML-схема. *Адресат: iiko.*

**Q4.** **Формат тела ошибки** HTTP 400: по полям (`{ "errors": { "bik": "..." } }`) или общий текст? *Адресат: Яндекс.*

**Q5.** **Формат контактных данных** в API: `full_name` (одна строка) или `first_name` + `last_name` (раздельно)? Ожидается YAML-схема. *Адресат: iiko.*

**Q6.** **DaData-интеграция** для подтягивания данных по ИНН — чья сторона (Яндекс или iiko)? *Адресат: Яндекс.*

**Q7.** **Обновлённая YAML-схема** API от Яндекса. *Адресат: Яндекс.*

## Часть 18. Резюме

Давайте подведу итоги.

**Что делаем:** добавляем OAuth 2.0 через Яндекс ID для автоматизации онбординга мерчантов.

**Ключевые решения:**
- Server-to-server архитектура — все вызовы Vendor Cabinet API через backend, Bearer-токен не на фронтенде.

- Полная обратная совместимость — ручной режим работает всегда, плагин обновления не требует.
- Два контура авторизации изолированы — OAuth (Bearer) и платёжный (Api-Key + User-Token) живут независимо.
- Минимальные изменения в существующем — два поля (`source`, `merchant_id`) в одной таблице.

**Объём работы:**
- **Frontend:** блок авторизации, две формы, экран статусов, интеграция с YaAuthSuggest SDK.
- **Backend:** 13 новых endpoints (4 OAuth + 9 proxy), 5 новых серверных компонентов, 3 миграции БД.
- **Плагин:** без изменений.
