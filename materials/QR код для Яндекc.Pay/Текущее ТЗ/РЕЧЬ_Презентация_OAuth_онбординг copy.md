## Часть 1. Что мы делаем и зачем

Ребят, давайте начнём с верхнего уровня. Мы дорабатываем панель Яндекс.Пэй в iikoWeb — автоматизируем подключение мерчантов через OAuth 2.0.

**Проблема сейчас (AS-IS):** Чтобы подключить ресторан к Яндекс.Пэй, ресторатор выполняет 4 ручных шага:

1. Заходит в личный кабинет Яндекс.Пэй
2. Копирует User Token вручную
3. Вставляет его в настройки iikoWeb
4. Вручную выбирает QR-табличку

Это занимает часы, а то и дни, и порождает ошибки при ручном копировании токенов.

**Что будет (TO-BE):** Автоматизированный процесс из 7 логических этапов:

1. **OAuth-авторизация** — кнопка «Войти с Яндекс ID» в iikoWeb
2. **Получение списка организаций** (Partner ID) через Vendor Cabinet API (`GET /partners`)
3. **Создание или выбор организации** — указание реквизитов, прежде всего ИНН (`POST /partners`)
4. **Подача заявки на подключение ресторана** (Merchant) — одна заявка = один ресторан, с указанием MCC-кода, реквизитов, контактов (`POST /merchants/register`)
5. **Ожидание обработки** (ориентировочно ~2 дня) + поллинг статуса (`GET /merchants/{id}/status`): `processing` → `active` / `failed`
6. **Получение User Token** автоматически после активации мерчанта (`POST /merchants/{id}/usertoken`)
7. **Маппинг QR-табличек** (POS ID) на кассы iikoFront (только на стороне iiko)

**Бизнес-выгоды:**
- Сокращение времени подключения мерчанта до нескольких минут
- Исключение ошибок ручного копирования токенов
- Единый интерфейс управления подключениями в iikoWeb
- Автоматический мониторинг статуса заявок

## Часть 2. Какие компоненты затрагиваем

Важный момент — мы не трогаем всё подряд. Есть чёткое разграничение:

### Затрагиваемые компоненты

| Компонент | Что делаем |
|-----------|-----------|
| **iikoWeb (панель Яндекс.Пэй)** | Основной компонент доработки. Добавляем OAuth-авторизацию через Яндекс ID, меняем UI панели администрирования |

- **Мобильные приложения** — не затрагиваются

## Часть 3. Иерархия сущностей — «кто есть кто»

Прежде чем погружаться в детали, давайте разберёмся с иерархией сущностей в Яндекс.Пэй. Это важно понимать для всей последующей логики:

```
Пользователь (Яндекс ID)
 └── Partner ID (организация / юридическое лицо)
      └── Merchant ID (магазин / ресторан)
           └── POS ID (QR-таблички, привязаны к кассам iiko)
```

### 3.1. Partner (Организация / Юридическое лицо)

Партнёр — это юрлицо (ООО, ИП). **Идентифицируется по ИНН**. Создаётся через `POST /partners`. При создании обязательно только поле `tax_ref_number` (ИНН) — остальные данные Яндекс подтягивает автоматически через DaData.

**Все поля сущности Partner (из Vendor Cabinet API):**

| Поле API | Тип | Обяз. | Описание | Откуда берётся |
|----------|-----|-------|----------|----------------|
| `partner_id` | VARCHAR(255) | — | ID партнёра в Яндекс.Пэй | Генерируется Яндексом при создании |
| `name` | String | Нет | Название организации | Ввод пользователя или DaData |
| `registration_data.tax_ref_number` | String(10-12) | **Да** | ИНН (10 цифр — юрлицо, 12 цифр — ИП). **Ключ идемпотентности** — если ИНН уже есть, API вернёт существующего партнёра | Ввод пользователя |

| `registration_data.legal_address` | String | Нет | Юридический адрес | Ввод пользователя или DaData |

**Итого:** 1 обязательное поле (`tax_ref_number`), 15 опциональных.

### 3.2. Merchant (Торговая точка / Ресторан)

Мерчант — это конкретный ресторан или торговая точка. Привязан к Partner. Создаётся через `POST /merchants/register`. Имеет статус регистрации, который меняется по мере обработки Яндексом.

**Все поля заявки на мерчанта (из Vendor Cabinet API):**

| Поле API | Тип | Обяз. | Описание | Откуда берётся |

| `merchant.address` | String | **Да** | Физический адрес торговой точки | Предзаполняется из iiko (уровень RMS) |
| `onboarding_data.mcc` | String(4) | **Да** | MCC-код — категория бизнеса из справочника `GET /mcc-codes`. По умолчанию `"5812"` (Рестораны) | Dropdown в UI, источник — `GET /mcc-codes` |
| `bank_details.bik` | String(9) | **Да** | БИК банка (ровно 9 цифр) | Ввод пользователя |

| `bank_details.correspondent_account` | String(20) | **Да** | Корреспондентский счёт (ровно 20 цифр) | Ввод пользователя |

| `communication_contact.phone` | String(+7XXXXXXXXXX) | **Да** | Телефон контакта | Ввод пользователя |
| `communication_contact.last_name` | String | **Да** | Фамилия контакта | Ввод пользователя |
| `communication_contact.first_name` | String | **Да** | Имя контакта | Ввод пользователя |
| `communication_contact.middle_name` | String | Нет | Отчество контакта | Ввод пользователя |
| `poses_count` | Integer | Нет | Количество QR-табличек (касс) для мерчанта | Количество выбранных чекбоксов касс в UI (макс. 10) |

**Статусы мерчанта** (поле `registration_status` в ответах API):

| Статус | Описание | Следующий шаг |
|--------|----------|---------------|
| `processing` | Заявка на рассмотрении у Яндекса (~2 дня) | Поллинг `GET /merchants/{id}/status`, не чаще 1 раз в 10 мин |
| `active` | Заявка одобрена, мерчант активен | Автоматически запрашиваем User Token (`POST /merchants/{id}/usertoken`) |
| `failed` | Заявка отклонена Яндексом | Показываем «Отклонено», можно подать повторно (новый `merchant_id`) |

### 3.3. POS (QR-табличка)

| Поле API | Описание |
|----------|----------|
| `pos_id` | ID QR-таблички в системе Яндекс.Пэй. Привязывается к терминалу iikoFront |
| `merchant_id` | Ссылка на мерчанта, которому принадлежит табличка |

**Ограничение:** максимум 10 QR-табличек (POS) на один Merchant ID. Если касс больше 10 — подаётся вторая заявка (создаётся новый Merchant ID).

## Часть 4. Два независимых контура авторизации

Это ключевой архитектурный момент, чтобы сразу не было путаницы. В системе работают **два полностью независимых контура авторизации**:

| # | Контур | Механизм | Заголовок запроса | API | Base URL (прод) | Base URL (тест) |
|---|--------|----------|-------------------|-----|-----------------|-----------------|
| 1 | **OAuth 2.0 (новый)** | Bearer-токен | `Authorization: Bearer <access_token>` | Vendor Cabinet API | `https://console.pay.yandex.net/api/vendor/v1` | `https://test.console.pay.yandex.net/api/vendor/v1` |
| 2 | **Api-Key + User-Token (существующий)** | Статические ключи | `Api-Key: <key>` + `User-Token: <token>` | Платежное API | `https://api.qr.yandex.net/api/v1/s1` | `https://tester.newpay.pro/api/v1/s1` |

Контуры полностью изолированы:
- **OAuth (контур 1)** — используется для управления организациями, мерчантами, выпуска User Token. Токен хранится в серверной сессии iikoWeb Backend
- **Api-Key + User-Token (контур 2)** — используется плагином iikoFront для платежных операций (оплата, возврат, отмена). Api-Key настраивается в XML-конфиге плагина, User Token хранится в БД iiko

Если OAuth упадёт — оплата на кассах продолжит работать. Доработка OAuth **не затрагивает** существующий механизм Api-Key / User-Token.

## Часть 5. OAuth 2.0 — как работает авторизация

### 5.1. Регистрация OAuth-приложения (однократно)

Первым делом нам нужно зарегистрировать OAuth-приложение в Яндекс. Это делается однократно нами (разработчиками) на https://oauth.yandex.ru/.

**Параметры регистрации:**

| Параметр | Значение | Где хранится | Примечание |
|----------|----------|-------------|-----------|
| `client_id` | Выдаётся Яндексом | Backend (конфиг) + транзитно в URL авторизации | Публичный параметр, можно передавать на фронтенд |
| `client_secret` | Выдаётся Яндексом | Backend (конфиг) **ТОЛЬКО** | **НИКОГДА** не попадает на фронтенд, не логируется |
| `redirect_uri` | `https://{iiko-web-domain}/api/oauth/callback` | Backend (конфиг) | Один на весь SaaS. Должен **точно** совпадать с указанным при регистрации |
| Тип приложения | «Для доступа к API» | Консоль oauth.yandex.ru | Определяет доступные grant types |
| Тип токена | **Продлеваемый** (рекомендуется) | Консоль oauth.yandex.ru | TTL не менее 1 года, продлевается при каждой авторизации |
| Scopes | **[?] Уточнить при регистрации** | Консоль oauth.yandex.ru | Конкретные scopes для Vendor Cabinet API не раскрыты в документации |

> **Открытый вопрос (Q1):** Конкретные scopes для Vendor Cabinet API пока не указаны в документации Яндекса. API возвращает HTTP 403 «Недостаточно прав (scope)» при отсутствии нужного scope. Уточним при регистрации приложения или у менеджера Яндекс.Пэй.

### 5.2. Authorization Code Flow — пошагово

Используем стандартный Authorization Code Flow:

| Шаг | Кто | Что происходит |
|-----|-----|---------------|
| 1 | Пользователь | Нажимает «Войти с Яндекс ID» в iikoWeb |
| 2 | iikoWeb Frontend | Перенаправляет браузер на `GET https://oauth.yandex.ru/authorize` с параметрами (см. ниже) |
| 3 | Яндекс ID | Показывает форму логина (popup или redirect) |
| 4 | Пользователь | Вводит учётные данные, подтверждает доступ |
| 5 | Яндекс ID | Перенаправляет браузер обратно на `redirect_uri` с параметрами `code` и `state` |
| 6 | iikoWeb Backend | Обменивает `code` на токены — server-to-server запрос `POST https://oauth.yandex.ru/token` |
| 7 | iikoWeb Backend | Сохраняет `access_token` и `refresh_token` в серверной сессии |
| 8 | iikoWeb Frontend | Показывает индикатор «Авторизован: user@yandex.ru» |

#### 5.2.1. Параметры запроса `GET /authorize`

```
GET https://oauth.yandex.ru/authorize?
    response_type=code
    &client_id={client_id}
    &redirect_uri={redirect_uri_encoded}
    &state={random_csrf_token}
```

| Параметр | Обяз. | Тип | Значение | Описание |
|----------|-------|-----|----------|----------|
| `response_type` | Да | Enum | `code` (фиксированное) | Тип ответа — авторизационный код |
| `client_id` | Да | String | ID приложения iiko в Яндекс | Из регистрации OAuth-приложения (п. 5.1) |
| `redirect_uri` | Да | String(URL) | `https://{iiko-web-domain}/api/oauth/callback` | Куда Яндекс вернёт пользователя. Должен **точно** совпадать с зарегистрированным |
| `state` | Да | String | Случайная строка (криптографически стойкая) | **CSRF-защита.** Backend генерирует (UUID v4 / crypto.randomBytes), сохраняет в сессии, проверяет при callback |

#### 5.2.2. Параметры запроса `POST /token` (обмен code → token)

```
POST https://oauth.yandex.ru/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code={authorization_code}
&client_id={client_id}
&client_secret={client_secret}
```

| Параметр | Обяз. | Тип | Описание | Откуда |
|----------|-------|-----|----------|--------|
| `grant_type` | Да | Enum | `authorization_code` (фиксированное) | Константа |
| `code` | Да | String | Авторизационный код от Яндекс ID. **Короткоживущий** (~30 сек) | Из redirect-ответа Яндекса (шаг 5) |
| `client_id` | Да | String | ID приложения iiko | Из конфига backend |
| `client_secret` | Да | String(secret) | Секрет приложения. **Только server-to-server** | Из конфига backend |

> **Важно:** Этот запрос делается **только** backend'ом (server-to-server). Браузер никогда не видит `client_secret`.

#### 5.2.3. Ответ `POST /token` (успех)

```json
{
  "access_token": "y0_AgAAAABx7hGc...",
  "refresh_token": "1:GN3s...",
  "token_type": "bearer",
  "expires_in": 31536000
}
```

| Поле ответа | Тип | Описание | Куда сохраняется |
|-------------|-----|----------|------------------|
| `access_token` | String | Bearer-токен для Vendor Cabinet API. Передаётся в заголовке `Authorization: Bearer <token>` | Серверная сессия iikoWeb Backend |
| `refresh_token` | String | Токен для обновления `access_token` без повторной авторизации | Серверная сессия iikoWeb Backend |
| `token_type` | String | Всегда `"bearer"` (RFC 6750) | Не хранится (константа) |
| `expires_in` | Integer | Время жизни `access_token` в секундах (обычно 31536000 = 1 год) | Для проверки TTL |

#### 5.2.4. Обновление токена (`POST /token` с refresh)

```
POST https://oauth.yandex.ru/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token={refresh_token}
&client_id={client_id}
&client_secret={client_secret}
```

| Параметр | Обяз. | Тип | Описание | Откуда |
|----------|-------|-----|----------|--------|
| `grant_type` | Да | Enum | `refresh_token` (фиксированное) | Константа |
| `refresh_token` | Да | String | Ранее полученный refresh_token | Из серверной сессии backend |
| `client_id` | Да | String | ID приложения iiko | Из конфига backend |
| `client_secret` | Да | String(secret) | Секрет приложения | Из конфига backend |

Ответ — идентичен п. 5.2.3 (новая пара `access_token` + `refresh_token`).

### 5.3. Ключевые моменты безопасности OAuth

- **Параметр `state` (CSRF-защита):** Backend генерирует криптографически стойкое случайное значение (UUID v4 / crypto.randomBytes), сохраняет в серверной сессии, передаёт как query-параметр в URL авторизации. При callback — сравнивает значение из ответа с сохранённым. При несовпадении — запрос отклоняется, инцидент логируется с меткой `[SECURITY INCIDENT]`
- **`client_secret`** — хранится **только** на backend (переменная окружения или конфиг-файл). Передаётся **только** в server-to-server вызовах к `POST https://oauth.yandex.ru/token`. Никогда не попадает на фронтенд. Не логируется
- **Все вызовы к Vendor Cabinet API** — исключительно server-to-server через backend. Браузер не имеет доступа к Bearer-токену и не выполняет прямых запросов к API Яндекс.Пэй
- **`access_token` и `refresh_token`** — хранятся в серверной сессии iikoWeb Backend (привязаны к пользовательской сессии). Рекомендуется шифрование at rest. **Не в БД** — при завершении сессии токены удаляются, нужна повторная авторизация

### 5.4. Управление токенами

| Аспект | Описание |
|--------|----------|
| **TTL access_token** | Определяется Яндексом (обычно ~1 год = 31536000 сек для продлеваемых токенов) |
| **Refresh-рекомендация** | Обновлять долгоживущие токены раз в 3 месяца (рекомендация Яндекса) |
| **При 401 от Vendor Cabinet API** | Backend пытается refresh через `POST /token` с `grant_type=refresh_token`. Если `refresh_token` тоже невалиден (`invalid_grant`) → очистить оба токена → вернуть фронтенду «не авторизован» → предложить повторный вход |
| **Отзыв пользователем** | Пользователь может отозвать доступ через https://id.yandex.ru/personal/data-access или сменой пароля. API вернёт 401 → обработка стандартная |
| **Хранение токенов** | Серверная сессия (НЕ БД). При завершении сессии — нужна повторная OAuth-авторизация |

### 5.5. Кнопка «Выйти»

Когда пользователь нажимает «Выйти»:
- Backend вызывает `POST /api/oauth/logout` → очищает `access_token` и `refresh_token` из серверной сессии
- UI переключается в состояние «Не авторизован» (кнопка «Войти с Яндекс ID»)
- Сессия Яндекс ID в браузере **сохраняется** — iiko не управляет сессией Яндекса
- Повторное «Войти с Яндекс ID» может пройти без ввода пароля, если сессия Яндекса в браузере ещё активна
- **Критически важно:** «Выйти» очищает **только** Bearer-токен. Ранее полученные User Token и привязки QR-табличек **продолжают работать** на кассах

### 5.6. Обработка ошибок OAuth

| Ситуация | Параметр ошибки | Действие iiko | Сообщение пользователю |
|----------|----------------|---------------|------------------------|
| Пользователь отменил авторизацию | `error=access_denied` | Закрыть popup, показать кнопку «Войти» | «Авторизация отклонена. Для подключения необходимо предоставить доступ» |
| Невалидный `state` (CSRF-атака) | — | Отклонить callback, логировать `[SECURITY INCIDENT]` | «Ошибка безопасности. Повторите попытку» |
| Истечение authorization code (~30 сек) | `error=invalid_grant` | Перенаправить на повторную авторизацию | «Время авторизации истекло. Повторите попытку» |
| Popup заблокирован браузером | `window.open` вернул `null` | Определить блокировку, показать подсказку | «Разрешите всплывающие окна для данного сайта» |
| **Таймаут callback (5 мин)** | — | Закрыть popup по таймеру | «Время ожидания авторизации истекло» |
| Некорректный `redirect_uri` | Ошибка на стороне Яндекса | Пользователь не вернётся на iiko | Ошибка конфигурации, требует fix на нашей стороне |

## Часть 7. Полный пользовательский сценарий (Happy Path)

Теперь давайте пройдём весь путь пользователя от начала до конца — все 17 шагов с указанием точных API-вызовов:

### Шаг 1: Открытие раздела
Пользователь открывает раздел «Интеграции» → «Яндекс.Пэй» в iikoWeb. В верхней части DetailPanel — блок авторизации с кнопкой «Войти с Яндекс ID».

### Шаг 2: Нажатие «Войти с Яндекс ID»
Frontend вызывает `GET /api/oauth/authorize` на backend iiko. Backend:
- Генерирует `state` (криптографически стойкая случайная строка)
- Сохраняет `state` в серверной сессии
- Формирует URL: `https://oauth.yandex.ru/authorize?response_type=code&client_id={client_id}&redirect_uri={redirect_uri}&state={state}`
- Возвращает URL → Frontend открывает popup или redirect

### Шаг 3: Авторизация на Яндексе
Пользователь вводит логин/пароль на Яндекс ID, подтверждает доступ.

### Шаг 4 (авто): Callback и обмен code на токен
Яндекс перенаправляет на `redirect_uri?code={authorization_code}&state={state}`. Backend:
1. Проверяет `state` — сравнивает с сохранённым в сессии. При несовпадении → 400, инцидент в лог
2. Вызывает `POST https://oauth.yandex.ru/token` с `grant_type=authorization_code`, `code`, `client_id`, `client_secret`
3. Получает `{ access_token, refresh_token, token_type: "bearer", expires_in: 31536000 }`
4. Сохраняет `access_token` и `refresh_token` в серверной сессии

### Шаг 5 (авто): Загрузка организаций
Сразу после авторизации backend вызывает `GET /partners` (Vendor Cabinet API) с заголовком `Authorization: Bearer {access_token}`.
- **Если организаций нет** → кнопка «Создать организацию»
- **Если одна** → выбирается автоматически
- **Если несколько** → список для выбора

### Шаг 6: Нажатие «Создать организацию»
Открывается форма (модальное окно или раскрывающаяся секция).

### Шаг 7: Заполнение и отправка формы организации
Пользователь заполняет ИНН (обязательное) + опционально остальные 14 полей (см. п. 3.1). Нажимает «Создать» → backend вызывает `POST /partners` с телом `{ registration_data: { tax_ref_number: "7707123456", ... }, contact: { ... } }`.

Ответ мгновенный (синхронная операция): `200 OK` с `partner_id`. Если ИНН уже есть — возвращается существующий партнёр (идемпотентность).

### Шаг 8 (авто): Загрузка мерчантов
Backend вызывает `GET /merchants?partner_id={partner_id}` — получает список мерчантов. Для новой организации список пуст → кнопка «Подать заявку на подключение».

### Шаг 9: Подготовка заявки
Пользователь:
- Выбирает ресторан из дерева iiko (dropdown)
- Система предзаполняет название и адрес из iiko

### Шаг 10: Заполнение и отправка заявки
Пользователь заполняет:
- MCC-код (dropdown из `GET /mcc-codes`, по умолчанию «5812 — Рестораны»)
- БИК (9 цифр), расчётный счёт (20 цифр), корр. счёт (20 цифр)
- Email, телефон, ФИО контакта
- Выбирает кассы чекбоксами (макс. 10 на заявку)

Нажимает «Подать заявку» → backend вызывает `POST /merchants/register` с полным телом (см. п. 6.5) → `200 OK` → `{ merchant_id, registration_status: "processing" }` → Toast «Заявка подана».

> **Одна заявка = один ресторан** — MVP-ограничение.

### Шаг 11 (авто): Поллинг статуса
Backend запускает поллинг `GET /merchants/{merchant_id}/status`:
- Не чаще 1 раза в 10 минут
- Дополнительно — при каждом открытии страницы
- Ожидаемое время обработки Яндексом — ~2 дня

### Шаг 12 (авто): Одобрение заявки
Поллинг возвращает `registration_status: "active"` → статус в UI меняется на «Подключен» (зелёный) → поллинг прекращается.

### Шаг 13 (авто): Создание User Token
Backend **автоматически** вызывает `POST /merchants/{merchant_id}/usertoken` → получает `{ token_id, token_value: "d8b3c9e..." }`.

### Шаг 14 (авто): Сохранение User Token в БД iiko
Backend **программно** вызывает существующий endpoint `POST /api/yandex-pay/codes` с телом:
```json
{
  "storeId": "<ID ресторана iiko — из таблицы YandexPayMerchantMapping>",
  "code": "<token_value — из ответа POST /merchants/{id}/usertoken>",
  "source": "oauth",
  "merchant_id": "<merchant_id>"
}
```
Поле «Ключ» в UI заполняется автоматически, становится readonly с подписью «Получен автоматически».

### Шаг 15 (авто): Загрузка QR-табличек
Backend загружает список QR-табличек через существующие endpoints `POST /api/yandex-pay/codes/{codeId}/refresh` и `GET /api/yandex-pay/accounts/{storeId}`.

### Шаг 16: Назначение QR-табличек на кассы
При OAuth — автоназначение: QR-коды от Яндекса обезличены (не привязаны к конкретным кассам), поэтому iiko назначает их на кассы в произвольном порядке (те кассы, что были выбраны при подаче заявки). После авто-назначения пользователь может переназначить вручную через dropdown.

Привязка сохраняется **мгновенно** (без кнопки Save) через `POST /api/yandex-pay/terminal-accounts` → Toast «Табличка назначена».

### Шаг 17: Выбор таблички по умолчанию
Через `POST /api/yandex-pay/codes/{storeId}/default-account` → default account установлен → ресторан полностью подключён, оплата готова к работе.

> **Шаги 4-5, 8, 11-15 выполняются системой автоматически**, без участия пользователя.

## Часть 8. Другие пользовательские сценарии

### 8.1. Организация уже существует (идемпотентность ИНН)

Если пользователь вводит ИНН, который уже зарегистрирован:
1. `POST /partners` вернёт **200 OK** с данными существующего партнёра (без создания дубликата)
2. Сообщение UI: «Организация с ИНН {inn} уже зарегистрирована в Яндекс.Пэй. Данные подтянуты автоматически»
3. `GET /merchants?partner_id={id}` загружает ранее зарегистрированные мерчанты (если есть)

### 8.2. Повторная авторизация (всё уже подключено)
Если пользователь повторно авторизуется — загружаются организации и мерчанты, все со статусом `"active"` показываются зелёным. Ранее настроенные интеграции (User Token, привязки QR-табличек) **продолжают работать** независимо от OAuth-авторизации.

### 8.3. Чужой аккаунт
Если пользователь авторизовался не под тем аккаунтом — `GET /partners` вернёт пустой список или организации другого аккаунта. Подсказка: «Нужной организации нет в списке? Нажмите "Выйти" и войдите под корректным аккаунтом».

### 8.4. Заявка отклонена Яндексом
`GET /merchants/{id}/status` возвращает `registration_status: "failed"` →
- Красная метка «Отклонено» → поллинг прекращается
- **API не возвращает причину** отклонения (нет поля `reason` / `failure_reason`)
- Рекомендация — обратиться в поддержку Яндекс.Пэй
- Кнопка «Подать повторно» → создаётся **новый** `merchant_id`

### 8.5. Больше 10 касс на одном ресторане
Лимит API Яндекса — 10 POS (касс) на один Merchant ID. Если касс больше 10:
1. Первая заявка: кассы 1-10 → `merchant_id_1`
2. Вторая заявка (тот же ресторан): кассы 11-20 → `merchant_id_2`
3. Результат: один RMS имеет несколько Merchant ID, каждый со своим набором QR-кодов

### 8.6. Смена OAuth-аккаунта
Кнопка «Выйти» (`POST /api/oauth/logout`) очищает **только** Bearer-токен из серверной сессии. **НЕ влияет на:**
- User Token в БД iiko (продолжает работать)
- Привязки QR-табличек (сохраняются)
- Работу плагина iikoFront (оплата продолжается)

Повторная авторизация под новым аккаунтом → `GET /partners` вернёт организации нового аккаунта.

### 8.7. Перепривязка QR-табличек
1. `GET /api/yandex-pay/terminal-accounts/{storeId}` — загрузить текущие привязки
2. Выбрать другую табличку в dropdown терминала
3. `POST /api/yandex-pay/terminal-accounts` — новая привязка мгновенно

### 8.8. Добавление новых QR-табличек
Новые POS создаются **только** в ЛК Яндекс.Пэй (Vendor Cabinet API не имеет метода создания POS). iiko только забирает список:
1. `POST /api/yandex-pay/codes/{codeId}/refresh` — обновить список
2. `GET /api/yandex-pay/accounts/{storeId}` — загрузить новые таблички
3. `POST /api/yandex-pay/terminal-accounts` — привязать к кассам

## Часть 10. Доработка iikoWeb Backend

### 10.1. Архитектурная роль

Backend выступает как **прокси-слой** между Frontend и Vendor Cabinet API. Frontend **никогда** не обращается к Vendor Cabinet напрямую. Все запросы server-to-server: backend берёт `access_token` из серверной сессии → добавляет `Authorization: Bearer <token>` → пересылает запрос.

```
Frontend (браузер)
    |
    |-- /api/oauth/*     --> OAuth Controller ---------> Яндекс ID (oauth.yandex.ru)
    |
    |-- /api/vendor/*    --> Vendor Proxy Controller --> Vendor Cabinet API (console.pay.yandex.net)
    |
    |-- /api/yandex-pay/* --> YandexPay Controller --> БД iiko (минимальные изменения)
    |
    |-- /api/internal/*  --> Internal Controller    --> БД iiko (без изменений)
```

### 10.2. Все 13 новых endpoints

#### Группа OAuth (/api/oauth/*)

| # | Метод | URL | Назначение | Request | Response |
|---|-------|-----|-----------|---------|----------|
| A | `GET` | `/api/oauth/authorize` | Инициирование OAuth — генерация `state`, формирование URL | — | `{ url: "https://oauth.yandex.ru/authorize?..." }` |
| B | `POST` | `/api/oauth/callback` | Обработка callback: проверка `state`, обмен `code` на `access_token` | `{ code, state }` | `{ access_token, refresh_token, expires_in }` (в сессии) |
| C | `POST` | `/api/oauth/logout` | Выход — очистка `access_token` и `refresh_token` из сессии | — | `{ success: true }` |
| D | `GET` | `/api/oauth/status` | Проверка: авторизован ли пользователь | — | `{ authenticated: bool, email: string }` |

#### Группа Proxy (/api/vendor/*)

| # | Метод | URL | Назначение | Проксируется в |
|---|-------|-----|-----------|----------------|
| E | `GET` | `/api/vendor/partners` | Список организаций | `GET /partners` |
| F | `POST` | `/api/vendor/partners` | Создание организации | `POST /partners` |
| G | `POST` | `/api/vendor/merchants/register` | Подача заявки | `POST /merchants/register` |
| H | `GET` | `/api/vendor/merchants/{id}/status` | Статус мерчанта | `GET /merchants/{id}/status` |
| I | `GET` | `/api/vendor/merchants` | Список мерчантов | `GET /merchants?partner_id={id}` |
| J | `GET` | `/api/vendor/merchants/{id}/usertoken` | Список user_token | `GET /merchants/{id}/usertoken` |
| **K** | **`POST`** | **`/api/vendor/merchants/{id}/usertoken`** | **Создание user_token + автосохранение в БД** | `POST /merchants/{id}/usertoken` |
| L | `DELETE` | `/api/vendor/merchants/{id}/usertoken/{token_id}` | Удаление user_token | `DELETE /merchants/{id}/usertoken/{token_id}` |
| M | `GET` | `/api/vendor/mcc-codes` | Справочник MCC-кодов (с кешированием 24ч) | `GET /mcc-codes` |

### 10.3. Endpoint K — ключевой (автосохранение User Token)

Этот endpoint **не просто проксирует** — у него дополнительная логика:

1. Проксирует `POST /merchants/{merchant_id}/usertoken` к Vendor Cabinet API
2. Получает ответ: `{ token_id, token_value, ... }`
3. **Определяет `storeId`** через маппинг: `merchant_id` → `store_id` из таблицы `yandex_pay_merchant_mapping`
4. **Программно вызывает** существующий `POST /api/yandex-pay/codes` с телом:
   ```json
   {
     "storeId": "<store_id — из таблицы YandexPayMerchantMapping>",
     "code": "<token_value — из ответа Vendor Cabinet API>",
     "source": "oauth",
     "merchant_id": "<merchant_id>"
   }
   ```
5. Возвращает ответ Vendor Cabinet API на Frontend

### 10.4. Новые серверные компоненты

| Компонент | Ответственность |
|-----------|----------------|
| **OAuth Controller** | Endpoints A-D: authorize, callback, logout, status |
| **Vendor Proxy Controller** | Endpoints E-M: проксирование всех запросов к Vendor Cabinet API |
| **OAuth Token Service** | Обмен `code` → `token`, refresh, хранение в сессии, валидация TTL |
| **Vendor Cabinet Client** | HTTP-клиент для server-to-server запросов. Подставляет `Authorization: Bearer <token>`. Обрабатывает 401 (refresh) |
| **MCC Cache Service** | In-memory кеш справочника MCC-кодов (TTL = 24 часа) |

### 10.5. Изменения в существующих endpoints

Все существующие endpoints `/api/yandex-pay/*` **сохраняют текущее поведение**. Единственное изменение:

**`POST /api/yandex-pay/codes`** — добавляются два поля:

| Новое поле | Тип | Default | Описание | Когда заполняется |
|------------|-----|---------|----------|-------------------|
| `source` | VARCHAR(20), NOT NULL | `"manual"` | Источник создания: `"manual"` (ручной ввод) или `"oauth"` (через OAuth) | При OAuth — `"oauth"`, при ручном вводе — `"manual"` |
| `merchant_id` | VARCHAR(255), NULL | `NULL` | ID мерчанта Яндекс.Пэй | Заполняется только при `source = "oauth"` |

**Pull-запрос от плагина iikoFront** (`POST /api/internal/yandex-pay/code`) — **без изменений**. Плагин не знает и не должен знать, как был получен токен.

### 10.6. Защита от SSRF

Backend проксирует запросы **только** к whitelisted хостам:
- `oauth.yandex.ru` — OAuth-авторизация
- `console.pay.yandex.net` — Vendor Cabinet API (прод)
- `test.console.pay.yandex.net` — Vendor Cabinet API (тест)

URL для каждого endpoint формируется на backend из конфигурации, **а не** из данных фронтенда.

### 10.7. Кеширование

| Endpoint | Стратегия | TTL | Обоснование |
|----------|-----------|-----|-------------|
| `GET /mcc-codes` | In-memory cache | 24 часа | Справочник статичный. Fallback: stale-кеш при недоступности API |
| Все остальные | Без кеширования | — | Данные могут измениться в ЛК Яндекс |

### 10.8. Логирование

**Что логировать:** факт начала/завершения авторизации, HTTP-статусы ответов, коды ошибок, `account_id` пользователя, факт обновления/отзыва токена, создание/изменение партнёра/мерчанта, изменение статуса мерчанта, CSRF-ошибки (`[SECURITY INCIDENT]`), timestamp.

**Что НЕ логировать:** `client_secret`, `access_token`, `refresh_token`, `authorization_code`, значение `state`, User Token (`token_value`), полные тела ответов с токенами, персональные данные (ИНН, адреса) в открытом виде.

### 10.9. Обработка ошибки 401

При получении HTTP 401 от Vendor Cabinet API backend выполняет:

1. Попытка обновить `access_token` через `POST https://oauth.yandex.ru/token` с `grant_type=refresh_token` (п. 5.2.4)
2. Если refresh успешен → повторить исходный запрос с новым `access_token`
3. Если `refresh_token` тоже невалиден (ответ `invalid_grant`) → очистить оба токена из сессии
4. Вернуть фронтенду статус «не авторизован» → предложить повторную OAuth-авторизацию

## Часть 12. Обратная совместимость

Это критически важная часть — мы **ничего не ломаем**.

### 12.1. Основные принципы

- **Аддитивность:** OAuth добавляет новые endpoints, UI-элементы и таблицы. Существующее не удаляется
- **Параллельность:** Оба режима (ручной и OAuth) доступны одновременно. Разные рестораны могут использовать разные способы
- **Прозрачность для плагина:** Плагин iikoFront работает одинаково. Pull-запрос `POST /api/internal/yandex-pay/code` возвращает токен из БД — без различия между `source = "manual"` и `source = "oauth"`
- **Независимость контуров:** Отзыв OAuth-токена не влияет на работу плагина, если User Token уже получен и сохранён

### 12.2. Что НЕ нужно делать существующим клиентам

- Переустанавливать плагин iikoFront
- Перенастраивать iikoOffice
- Удалять старый User Token
- Перепривязывать QR-таблички
- Менять конфигурацию Api-Key

### 12.3. Feature Toggle

OAuth-функционал управляется параметром конфигурации `YandexPayOAuthEnabled`:

| Значение toggle | Блок OAuth в UI | Ручной ввод | `/api/oauth/*` | `/api/vendor/*` | Плагин iikoFront |
|-----------------|----------------|-------------|----------------|-----------------|------------------|
| `false` (по умолчанию) | **Скрыт** | Доступен | HTTP 404 | HTTP 404 | Работает |
| `true` (не авторизован) | Видна кнопка «Войти» | Доступен | Активны | HTTP 401 | Работает |
| `true` (авторизован) | Полная панель | Доступен | Активны | Активны | Работает |

### 12.4. Rollback-стратегия

**Уровень 1 — Toggle off (минуты):**
- `YandexPayOAuthEnabled = false`
- OAuth скрыт, ручной режим работает, все токены и привязки сохранены

**Уровень 2 — Полный rollback (десятки минут):**
- Откат версии iikoWeb
- Обратные миграции БД:
  ```sql
  DROP TABLE IF EXISTS yandex_pay_partner_mapping;
  DROP TABLE IF EXISTS yandex_pay_merchant_mapping;
  ALTER TABLE yandex_pay_code DROP COLUMN IF EXISTS source;
  ALTER TABLE yandex_pay_code DROP COLUMN IF EXISTS merchant_id;
  ```
- Все User Token продолжают работать (поле `code` не затронуто)
- Данные маппинга утрачены → потребуется повторный онбординг через OAuth

## Часть 13. Ограничения MVP

Фиксирую ключевые ограничения, чтобы не было завышенных ожиданий:

| # | Ограничение | Пояснение |
|---|------------|-----------|
| 1 | **Одна заявка = один ресторан** | Batch-подача нескольких мерчантов одним запросом не реализуется |
| 2 | **Максимум 10 касс на один Merchant ID** | Ограничение API Яндекса. Если больше — вторая заявка |
| 3 | **Один OAuth-аккаунт на сессию** | Для управления разными организациями — выход и повторный вход |
| 4 | **Нет фонового обновления статусов** | Обновление при открытии страницы или поллинг не чаще 1 раз в 10 мин |
| 5 | **Нет глубокой синхронизации** | Изменения в ЛК Яндекс.Пэй не отражаются в iiko автоматически |
| 6 | **QR-маппинг только на стороне iiko** | Яндекс не хранит привязку «табличка ↔ касса» |
| 7 | **Добавление новых QR-табличек — только через ЛК Яндекс** | Vendor Cabinet API не имеет метода создания POS. iikoWeb их только забирает |
| 8 | **Api-Key не редактируется через OAuth** | OAuth-процесс получает только User Token. Api-Key настраивается в XML-конфиге плагина |
| 9 | **Управление токенами автоматическое** | UI для ручного управления `token_id` не реализуется |
| 10 | **Маппинг существующих мерчантов из ЛК Яндекс не входит в MVP** | Связывание ранее зарегистрированных мерчантов с ресторанами iiko не реализуется |
| 11 | **Касса iiko не имеет атрибута физического адреса** | Адрес доступен только на уровне RMS. Для кофе-поинтов адрес в заявке = адрес RMS |
| 12 | **OAuth-токены хранятся в серверной сессии** | При завершении сессии — повторная авторизация. Фоновое обновление не реализуется |
| 13 | **Частичные возвраты не поддерживаются** | Ограничение платежного API Яндекс (только полный возврат) |

## Часть 14. Обработка ошибок — ключевые сценарии

### Ошибки OAuth

| Ситуация | Действие | Сообщение |
|----------|---------|-----------|
| Пользователь отменил (`error=access_denied`) | Закрыть popup, показать кнопку «Войти» | «Авторизация отклонена. Для подключения необходимо предоставить доступ» |
| Невалидный `state` (CSRF) | Отклонить callback, логировать `[SECURITY INCIDENT]` | «Ошибка безопасности. Повторите попытку» |
| Истечение authorization code (`invalid_grant`) | Перенаправить на повторную авторизацию | «Время авторизации истекло. Повторите попытку» |
| Popup заблокирован (`window.open = null`) | Определить блокировку, показать подсказку | «Разрешите всплывающие окна для данного сайта» |
| **Таймаут callback (5 мин)** | Закрыть popup по таймеру | «Время ожидания авторизации истекло» |
| Некорректный `redirect_uri` | Пользователь не вернётся (ошибка на Яндексе) | Ошибка конфигурации, требует fix |

### Ошибки API (Vendor Cabinet)

| HTTP | Ситуация | Действие | Сообщение |
|------|----------|---------|-----------|
| 400 | Невалидные данные (ИНН, БИК, email и т.д.) | Подсветить невалидные поля, оставить форму открытой | Зависит от поля: «ИНН должен содержать 10 или 12 цифр», «БИК — 9 цифр» и т.д. |
| 401 | Токен истёк / отозван | Попытка refresh (п. 10.9). При неудаче → сброс авторизации | «Сессия авторизации истекла. Необходимо повторно войти через Яндекс ID» |
| 403 | Нет доступа (чужой ресурс или недостаточно scope) | Логировать. Предложить «Выйти» и войти под другим аккаунтом | «Нет доступа. Убедитесь, что авторизовались под корректным аккаунтом Яндекс» |
| 404 | Ресурс не найден (удалён в ЛК Яндекс) | Обновить маппинг в БД | «Объект не найден. Возможно, он был удалён» |
| 500 | Сервер Яндекс недоступен | Toast, **НЕ** повторять автоматически, логировать | «Сервис Яндекс.Пэй временно недоступен. Попробуйте позже» |

### Ошибка получения User Token

При переходе мерчанта в статус `"active"`, backend автоматически вызывает `POST /merchants/{id}/usertoken`. Если этот вызов упал:
- Retry: до 3 попыток, интервал 10 сек
- При неудаче: пользователю показывается сообщение «Не удалось автоматически получить токен. Попробуйте обновить страницу»

## Часть 15. Плагин iikoFront

Минимальные изменения. Плагин получает User Token из БД iiko через pull-запрос (`POST /api/internal/yandex-pay/code`). Этот endpoint **не меняется**. Способ получения токена (ручной или OAuth, `source = "manual"` / `"oauth"`) **прозрачен** для плагина — он просто получает значение `code` (= `token_value`).

**Что НЕ меняется:**
- Конфигурационный XML-файл плагина
- Api-Key (настраивается в XML, не через OAuth)
- Платежное API (операции оплаты, возврата, отмены)
- Обновление версии плагина **не требуется**

## Часть 18. Резюме для команды

### Что делаем:
Добавляем OAuth 2.0 авторизацию через Яндекс ID для автоматизации онбординга мерчантов в панели Яндекс.Пэй iikoWeb.

### Ключевые решения:
1. **Server-to-server архитектура** — все вызовы Vendor Cabinet API через backend, Bearer-токен не на фронтенде
2. **Feature toggle `YandexPayOAuthEnabled`** — можно включить/выключить без деплоя кода
3. **Полная обратная совместимость** — ручной режим работает всегда, плагин не требует обновления
4. **Два контура авторизации** — OAuth (Bearer для Vendor Cabinet) и платёжный (Api-Key + User-Token) изолированы друг от друга
5. **Минимальные изменения в существующем** — только два поля (`source`, `merchant_id`) в одной таблице `yandex_pay_code`
6. **Одна заявка = один ресторан** — MVP-ограничение для простоты

### Объём работы по компонентам:
- **iikoWeb Frontend:** блок авторизации, формы организации/мерчанта, экран статусов, интеграция с YaAuthSuggest SDK
- **iikoWeb Backend:** 13 новых endpoints (4 OAuth + 9 proxy), 5 новых компонентов, 3 миграции БД
- **Плагин iikoFront:** без изменений (получение токена через `POST /api/internal/yandex-pay/code` — прозрачно)

### Главный риск:
Зависимость от Яндекса — готовность Vendor Cabinet API, уточнение scopes, время обработки заявок (~2 дня), 11 открытых вопросов.
