## Часть 1. Что мы делаем и зачем

Ребят, давайте начнём с верхнего уровня. Мы дорабатываем панель Яндекс.Пэй в iikoWeb — автоматизируем подключение мерчантов через OAuth 2.0.

**Проблема сейчас (AS-IS):** Чтобы подключить ресторан к Яндекс.Пэй, ресторатор выполняет 4 ручных шага:

1. Заходит в личный кабинет Яндекс.Пэй
2. Копирует User Token вручную
3. Вставляет его в настройки iikoWeb
4. Вручную выбирает QR-табличку

Это занимает часы, а то и дни, и порождает ошибки при ручном копировании токенов.

**Что будет (TO-BE):** Автоматизированный процесс. Ресторатор нажимает кнопку «Войти с Яндекс ID», авторизуется — и дальше iiko автоматически получает доступ к Vendor Cabinet API, сама создаёт организацию, подаёт заявку, получает токен и привязывает QR-таблички к кассам. Время подключения — минуты вместо часов/дней.

**Бизнес-выгоды:**
- Сокращение времени подключения мерчанта до нескольких минут
- Исключение ошибок ручного копирования токенов
- Подача заявок на подключение ресторанов через единый интерфейс (одна заявка = один ресторан)
- Единый интерфейс управления подключениями в iikoWeb
- Автоматический мониторинг статуса заявок

## Часть 2. Какие компоненты затрагиваем

Важный момент — мы не трогаем всё подряд. Есть чёткое разграничение:

### Затрагиваемые компоненты

| Компонент | Что делаем |
|-----------|-----------|
| **iikoWeb (панель Яндекс.Пэй)** | Основной компонент доработки. Добавляем OAuth-авторизацию через Яндекс ID, меняем UI панели администрирования |
| **iikoWeb Backend** | Серверная часть OAuth 2.0 — обмен authorization code на токены, взаимодействие с Vendor Cabinet API, хранение токенов |
| **Плагин iikoFront** | Минимальные изменения — получение User Token через backend вместо локальной конфигурации |

### НЕ затрагиваемые компоненты

- **Платежное API Яндекс.Пэй** — операции оплаты, возврата, отмены — работает без изменений
- **iikoOffice** — не затрагивается
- **Мобильные приложения** — не затрагиваются

## Часть 3. Иерархия сущностей — «кто есть кто»

Прежде чем погружаться в детали, давайте разберёмся с иерархией сущностей в Яндекс.Пэй. Это важно понимать для всей последующей логики:

```
Пользователь (Яндекс ID)
 └── Partner ID (организация / юридическое лицо)
      └── Merchant ID (магазин / ресторан)
           └── POS ID (QR-таблички, привязаны к кассам iiko)
```

- **Partner** — это юрлицо (ООО, ИП). Идентифицируется ИНН. Содержит реквизиты: ИНН, ОГРН, КПП, адрес
- **Merchant** — это конкретный ресторан / торговая точка. Привязан к Partner. Имеет статусы: `processing`, `active`, `failed`
- **POS ID** — это QR-табличка. По сути — идентификатор кассы в системе Яндекс.Пэй. Привязывается к терминалу iikoFront

Один Partner может иметь несколько Merchant'ов. У одного Merchant'а может быть максимум 10 QR-табличек (POS). Если касс больше 10 — нужна вторая заявка (новый Merchant ID).

## Часть 4. Два независимых контура авторизации

Это ключевой архитектурный момент, чтобы сразу не было путаницы. В системе работают **два полностью независимых контура авторизации**:

| # | Контур | Механизм | API | Назначение |
|---|--------|----------|-----|------------|
| 1 | **OAuth 2.0 (новый)** | Bearer-токен (access_token) | Vendor Cabinet API | Управление партнёрами, мерчантами, выпуск User Token |
| 2 | **Api-Key + User-Token (существующий)** | Статические ключи в заголовках | Платежное API | Операции оплаты, возвраты, отмены |

Контуры полностью изолированы. Доработка OAuth **не затрагивает** существующий механизм Api-Key / User-Token, который плагин iikoFront использует для платежных операций. Если OAuth упадёт — оплата на кассах продолжит работать.

## Часть 5. OAuth 2.0 — как работает авторизация

### 5.1. Регистрация OAuth-приложения (однократно)

Первым делом нам нужно зарегистрировать OAuth-приложение в Яндекс. Это делается однократно нами (разработчиками) на https://oauth.yandex.ru/. В результате получаем пару `client_id` / `client_secret`, которая используется для всех инсталляций iikoWeb.

Важный момент — `redirect_uri` будет один на весь SaaS, формат: `https://{iiko-web-domain}/api/oauth/callback`.

Тип токена рекомендуется **продлеваемый** — TTL не менее 1 года, продлевается при каждой авторизации.

> **Открытый вопрос (Q1):** Конкретные scopes для Vendor Cabinet API пока не указаны в документации Яндекса. Уточним при регистрации приложения.

### 5.2. Authorization Code Flow — пошагово

Теперь сам OAuth-поток. Используем стандартный Authorization Code Flow:

| Шаг | Кто | Что происходит |
|-----|-----|---------------|
| 1 | Пользователь | Нажимает «Войти с Яндекс ID» в iikoWeb |
| 2 | iikoWeb Frontend | Перенаправляет браузер на `GET https://oauth.yandex.ru/authorize` с параметрами `client_id`, `redirect_uri`, `response_type=code`, `state` |
| 3 | Яндекс ID | Показывает форму логина |
| 4 | Пользователь | Вводит учётные данные, подтверждает доступ |
| 5 | Яндекс ID | Перенаправляет браузер обратно на `redirect_uri` с `code` и `state` |
| 6 | iikoWeb Backend | Обменивает `code` на токены — server-to-server запрос `POST https://oauth.yandex.ru/token` с `client_secret` |
| 7 | iikoWeb Backend | Сохраняет `access_token` и `refresh_token` в сессии |
| 8 | iikoWeb Frontend | Показывает индикатор «Авторизован: user@yandex.ru» |

### 5.3. Ключевые моменты безопасности OAuth

- **Параметр `state` (CSRF-защита):** Backend генерирует криптографически стойкое случайное значение, сохраняет в сессии, передаёт в URL. При callback — проверяет. При несовпадении — запрос отклоняется, инцидент логируется
- **`client_secret`** — хранится **только** на backend. Передаётся **только** в server-to-server вызовах. Никогда не попадает на фронтенд
- **Все вызовы к Vendor Cabinet API** — исключительно server-to-server через backend. Браузер не имеет доступа к Bearer-токену и не выполняет прямых запросов к API Яндекс.Пэй
- **`access_token` и `refresh_token`** — хранятся в серверной сессии iikoWeb Backend. Рекомендуется шифрование at rest

### 5.4. Управление токенами

- **TTL access_token** — определяется Яндексом (обычно ~1 год для продлеваемых)
- **Refresh:** рекомендация Яндекса — обновлять долгоживущие токены раз в 3 месяца
- **При 401 от Vendor Cabinet API:** backend пытается refresh → если не получилось → сбрасывает авторизацию → предлагает пользователю повторный вход
- **Отзыв пользователем:** пользователь может отозвать доступ через https://id.yandex.ru/personal/data-access или сменой пароля. В этом случае API вернёт 401 → обработка стандартная

### 5.5. Кнопка «Выйти»

Когда пользователь нажимает «Выйти»:
- Backend очищает `access_token` и `refresh_token` из сессии
- UI переключается в состояние «Не авторизован»
- Сессия Яндекс ID в браузере **сохраняется** — iiko не управляет сессией Яндекса
- Повторное «Войти с Яндекс ID» может пройти без ввода пароля, если сессия Яндекса в браузере ещё активна

## Часть 6. Vendor Cabinet API — новый API от Яндекса

### 6.1. Общие сведения

| Параметр | Значение |
|----------|----------|
| Base URL (прод) | `https://console.pay.yandex.net/api/vendor/v1` |
| Base URL (тест) | `https://test.console.pay.yandex.net/api/vendor/v1` |
| Авторизация | `Authorization: Bearer <access_token>` |
| Content-Type | `application/json` |

### 6.2. Сводка методов

| # | Метод | Endpoint | Что делает |
|---|-------|----------|-----------|
| 1 | GET | /partners | Получить список организаций (партнёров) пользователя |
| 2 | POST | /partners | Создать нового партнёра (или получить существующего по ИНН — идемпотентность) |
| 3 | POST | /merchants/register | Создать заявку на подключение мерчанта (одна заявка = один ресторан) |
| 4 | GET | /merchants/{merchant_id}/status | Получить статус регистрации мерчанта (для поллинга) |
| 5 | GET | /merchants?partner_id={id} | Список мерчантов партнёра |
| 6 | GET | /merchants/{merchant_id}/usertoken | Список user_token (хеши) мерчанта |
| 7 | POST | /merchants/{merchant_id}/usertoken | Создать новый user_token для мерчанта |
| 8 | DELETE | /merchants/{merchant_id}/usertoken/{token_id} | Удалить user_token |
| 9 | GET | /mcc-codes | Справочник MCC-кодов |

### 6.3. Идемпотентность POST /partners

Важный момент: `POST /partners` идемпотентен по ИНН. Если организация с таким ИНН уже существует — API вернёт 200 OK с данными существующего партнёра, дубликат не создаётся. Это штатное поведение, не ошибка.

### 6.4. Ограничение: API не возвращает причину отклонения

Ответ `GET /merchants/{id}/status` содержит **только** `merchant_id`, `registration_status` и `poses[]`. Поля `reason` или `failure_reason` нет. Если заявка отклонена — мы просто показываем «Отклонено» и рекомендуем обратиться в поддержку Яндекс.Пэй.

## Часть 7. Полный пользовательский сценарий (Happy Path)

Теперь давайте пройдём весь путь пользователя от начала до конца — все 17 шагов:

### Шаг 1: Пользователь открывает раздел «Интеграции» → «Яндекс.Пэй»
В верхней части DetailPanel — блок авторизации с кнопкой «Войти с Яндекс ID».

### Шаг 2-4: OAuth-авторизация
Нажимает кнопку → popup/redirect на Яндекс → логин/пароль → подтверждение доступа → callback → backend обменивает `code` на `access_token` → UI показывает «Авторизован: user@yandex.ru».

### Шаг 5: Автоматическая загрузка организаций
Сразу после авторизации система вызывает `GET /partners`. Если организаций нет — кнопка «Создать организацию». Если одна — выбирается автоматически. Если несколько — список для выбора.

### Шаг 6-7: Создание организации
Если организации нет — пользователь заполняет форму. Обязательное поле — только ИНН. Яндекс сам подтягивает ОГРН, КПП и другие реквизиты по ИНН через DaData. Остальные поля опциональны. Нажимает «Создать» → `POST /partners` → организация создана мгновенно.

### Шаг 8: Автоматическая загрузка мерчантов
Система вызывает `GET /merchants?partner_id={id}`. Для новой организации список пуст — кнопка «Подать заявку на подключение».

### Шаг 9-10: Подача заявки на мерчанта
Пользователь:
- Выбирает ресторан из дерева iiko
- Выбирает MCC-код из dropdown (справочник `GET /mcc-codes`, по умолчанию «5812 — Рестораны»)
- Заполняет поля: название точки, физический адрес, БИК (9 цифр), расчётный счёт (20 цифр), корр. счёт (20 цифр), email, телефон, ФИО контакта
- Выбирает кассы чекбоксами (максимум 10 штук на одну заявку)
- Нажимает «Подать заявку» → `POST /merchants/register` → toast «Заявка подана»

> **Одна заявка = один ресторан** — это MVP-ограничение. Batch-подача нескольких мерчантов одним запросом не реализуется в текущей версии.

### Шаг 11: Поллинг статуса
Система запускает поллинг `GET /merchants/{id}/status`:
- Не чаще 1 раза в 10 минут
- Дополнительно — при каждом открытии страницы
- Ожидаемое время обработки Яндексом — около 2 дней

### Шаг 12: Одобрение заявки
Поллинг возвращает `registration_status: "active"` → статус в UI меняется на «Подключен» (зелёный) → поллинг прекращается.

### Шаг 13-14: Автоматическое получение User Token
Система **автоматически** вызывает `POST /merchants/{id}/usertoken` → получает `token_value` → сохраняет в БД iiko → поле «Ключ» заполняется автоматически. Пользователю **не нужно** копировать токен вручную.

### Шаг 15: Автоматическая загрузка QR-табличек
Система загружает список QR-табличек через существующие endpoints.

### Шаг 16: Назначение QR-табличек на кассы
Пользователь выбирает в dropdown терминала нужную табличку → привязка сохраняется **мгновенно** (без кнопки Save) → toast «Табличка назначена».

### Шаг 17: Выбор таблички по умолчанию
Default account выбран → ресторан полностью подключён, оплата готова к работе.

> **Шаги 4-5, 8, 11-15 выполняются автоматически**, без участия пользователя.

## Часть 8. Другие пользовательские сценарии

### 8.1. Организация уже существует
Если пользователь вводит ИНН, который уже зарегистрирован — `POST /partners` вернёт существующего партнёра (идемпотентность). Сообщение: «Организация с ИНН {inn} уже зарегистрирована в Яндекс.Пэй. Данные подтянуты автоматически». Дубликат не создаётся.

### 8.2. Повторная авторизация (всё уже подключено)
Если пользователь повторно авторизуется — загружаются организации и мерчанты, все со статусом «active» показываются зелёным. Ранее настроенные интеграции (User Token, привязки QR-табличек) **продолжают работать** независимо от OAuth-авторизации.

### 8.3. Чужой аккаунт
Если пользователь авторизовался не под тем аккаунтом — он просто не увидит нужной организации. Подсказка: «Нужной организации нет в списке? Нажмите "Выйти" и войдите под корректным аккаунтом».

### 8.4. Заявка отклонена Яндексом
Статус переходит в `failed` → красная метка «Отклонено» → поллинг прекращается → рекомендация обратиться в поддержку Яндекс.Пэй → можно подать повторную заявку (создаётся новый `merchant_id`).

### 8.5. Больше 10 касс на одном ресторане (кофе-поинты)
Лимит API Яндекса — 10 POS (касс) на один Merchant ID. Если касс больше 10 — пользователь подаёт вторую заявку на тот же ресторан. Создаётся новый Merchant ID с новым набором QR-кодов. У одного RMS может быть несколько Merchant ID.

### 8.6. Смена OAuth-аккаунта
Кнопка «Выйти» очищает **только** Bearer-токен. НЕ влияет на User Token и привязки QR-табличек. Ранее настроенные интеграции продолжают работать на кассах.

## Часть 9. Изменения в интерфейсе

### 9.1. Новый блок авторизации

Размещается в DetailPanel, **выше** поля «Ключ» (User Token). Имеет два состояния:

**Не авторизован:**
- Заголовок «Автоматическое подключение»
- Пояснительный текст
- Кнопка «Войти с Яндекс ID» (стилизация по гайдлайнам Яндекса — YaAuthSuggest SDK)
- Разделитель «или» — отделяет от ручного ввода ключа

**Авторизован:**
- Индикатор «Авторизован: {email}» (зелёный)
- Кнопка «Выйти» (красная)
- Селектор организации
- Список мерчантов и их статусы

### 9.2. Важно: ручной ввод остаётся

Блок ручного ввода ключа **остаётся доступным** ниже разделителя — это альтернативный способ подключения. Если User Token получен через OAuth — поле становится readonly с подписью «Получен автоматически». Кнопка «Очистить ключ» доступна всегда.

### 9.3. Форма создания организации

Модальное окно или раскрывающаяся секция. Обязательно **только** поле ИНН (10 или 12 цифр). Остальные 14 полей (ОГРН, КПП, юр. адрес, контакты и т.д.) — опциональны, Яндекс подтягивает данные по ИНН через DaData. Операция синхронная — ответ мгновенный, поллинг не нужен.

### 9.4. Форма подачи заявки на мерчанта

Содержит:
- Dropdown выбора ресторана из дерева iiko
- MCC-код (dropdown из справочника)
- Название торговой точки (предзаполняется из iiko)
- Физический адрес
- Банковские реквизиты: БИК (9 цифр), расчётный счёт (20 цифр), корр. счёт (20 цифр) — в одну строку (3 колонки)
- Контакт: email, телефон, ФИО
- Чекбоксы касс (максимум 10, уже подключённые — disabled)

### 9.5. Экран статуса заявок

Под блоком авторизации. Каждый мерчант — строка:
- Название | Статус | Дата подачи | Действия
- Статусы: «Обрабатывается» (жёлтый), «Подключен» (зелёный), «Отклонен» (красный)
- Кнопки: «Перейти в настройки» (active) | «Подать повторно» (failed)

### 9.6. Изменения в секции QR-табличек

| Элемент | Было | Стало |
|---------|------|-------|
| Поле «Ключ» | Ручной ввод | Автозаполнение после OAuth, readonly, подпись «Получен автоматически» |
| Кнопка «Сохранить» ключ | Ручное сохранение | Скрыта при автозаполнении |
| Кнопка «Очистить ключ» | Удаляет ключ | Остаётся, с предупреждением |
| Список QR-табличек | Ручная загрузка | Автоматическая после получения User Token |
| Привязка табличек | Dropdown | При OAuth — автоназначение на выбранные при заявке кассы |

### 9.7. Кнопка «Войти с Яндекс ID» — технические детали

Два варианта реализации (определяется на этапе разработки):
1. **YaAuthSuggest SDK** — рекомендуется Яндексом. SDK управляет popup и возвратом результата через postMessage
2. **Стандартный OAuth redirect** — `window.open` с ручной обработкой callback

## Часть 10. Доработка iikoWeb Backend

### 10.1. Архитектурная роль

Backend выступает как **прокси-слой** между Frontend и Vendor Cabinet API. Frontend **никогда** не обращается к Vendor Cabinet напрямую. Все запросы server-to-server: backend берёт `access_token` из сессии → добавляет `Authorization: Bearer <token>` → пересылает запрос.

```
Frontend (браузер)
    |
    |-- /api/oauth/*     --> OAuth Controller ---------> Яндекс ID (oauth.yandex.ru)
    |
    |-- /api/vendor/*    --> Vendor Proxy Controller --> Vendor Cabinet API
    |
    |-- /api/yandex-pay/* --> YandexPay Controller --> БД iiko (минимальные изменения)
    |
    |-- /api/internal/*  --> Internal Controller    --> БД iiko (без изменений)
```

### 10.2. Новые endpoints

#### Группа OAuth (/api/oauth/*)

| Метод | URL | Назначение |
|-------|-----|-----------|
| GET | /api/oauth/authorize | Генерация URL авторизации (state + redirect) |
| POST | /api/oauth/callback | Обработка callback от Яндекса (проверка state, обмен code на token) |
| POST | /api/oauth/logout | Выход — очистка токенов из сессии |
| GET | /api/oauth/status | Проверка: авторизован ли пользователь |

#### Группа Proxy (/api/vendor/*)

| Метод | URL | Назначение |
|-------|-----|-----------|
| GET | /api/vendor/partners | Список организаций |
| POST | /api/vendor/partners | Создание организации |
| POST | /api/vendor/merchants/register | Подача заявки на мерчанта |
| GET | /api/vendor/merchants/{id}/status | Статус регистрации |
| GET | /api/vendor/merchants | Список мерчантов |
| GET | /api/vendor/merchants/{id}/usertoken | Список user_token |
| POST | /api/vendor/merchants/{id}/usertoken | Создание user_token **(+ сохранение в БД iiko)** |
| DELETE | /api/vendor/merchants/{id}/usertoken/{token_id} | Удаление user_token |
| GET | /api/vendor/mcc-codes | Справочник MCC-кодов (с кешированием) |

### 10.3. Новые серверные компоненты

| Компонент | Ответственность |
|-----------|----------------|
| **OAuth Controller** | Endpoints авторизации: authorize, callback, logout, status |
| **Vendor Proxy Controller** | Проксирование запросов к Vendor Cabinet API |
| **OAuth Token Service** | Обмен code на token, refresh, хранение в сессии, валидация TTL |
| **Vendor Cabinet Client** | HTTP-клиент для server-to-server запросов к Vendor Cabinet API |
| **MCC Cache Service** | Кеширование справочника MCC-кодов (in-memory, TTL = 24 часа) |

### 10.4. Изменения в существующих endpoints

Все существующие endpoints `/api/yandex-pay/*` **сохраняют текущее поведение**. Единственное изменение:

**POST /api/yandex-pay/codes** — добавляются два поля:
- `source` — `"manual"` (ручной ввод) или `"oauth"` (через OAuth)
- `merchant_id` — заполняется при `source = "oauth"`

Pull-запрос от плагина iikoFront (`POST /api/internal/yandex-pay/code`) — **без изменений**. Плагин не знает и не должен знать, как был получен токен.

### 10.5. Обработка ошибки 401

При получении HTTP 401 от Vendor Cabinet API backend выполняет:

1. Попытка обновить `access_token` через `refresh_token`
2. Если `refresh_token` тоже невалиден — очистить оба токена из сессии
3. Вернуть фронтенду статус «не авторизован»
4. Предложить пользователю повторную OAuth-авторизацию

### 10.6. Защита от SSRF

Backend проксирует запросы **только** к whitelisted хостам:
- `oauth.yandex.ru` — OAuth-авторизация
- `console.pay.yandex.net` — Vendor Cabinet API (прод)
- `test.console.pay.yandex.net` — Vendor Cabinet API (тест)

URL для каждого endpoint формируется на backend из конфигурации, **а не** из данных фронтенда.

### 10.7. Кеширование

| Endpoint | Стратегия | TTL |
|----------|-----------|-----|
| GET /mcc-codes | In-memory cache | 24 часа |
| Все остальные | Без кеширования | — |

MCC-коды — статичные данные, обновляются редко. Если API недоступен — используем stale кеш. Если кеш пуст — ошибка.

### 10.8. Логирование

**Что логировать:** факт начала/завершения авторизации, HTTP-статусы, коды ошибок, идентификатор пользователя, факт обновления/отзыва токена, timestamp.

**Что НЕ логировать:** `client_secret`, `access_token`, `refresh_token`, `authorization_code`, значение `state`, User Token, полные тела ответов с токенами, персональные данные (ИНН, адреса) в открытом виде.

### 11.1. Новая таблица: `yandex_pay_partner_mapping`

Связь «ресторан iiko» ↔ «мерчант Яндекс.Пэй»:

| Поле | Тип | Описание |
|------|-----|----------|

| account_id | INT | ID аккаунта iikoWeb |

UNIQUE: `(account_id, store_id)` — один ресторан → один мерчант (в рамках аккаунта).

### 11.3. Изменения в существующей таблице `yandex_pay_code`

Добавляются два поля:
- `source` (VARCHAR(20), NOT NULL, DEFAULT `"manual"`) — источник создания
- `merchant_id` (VARCHAR(255), NULL) — связь с мерчантом (заполняется для OAuth)

DEFAULT `"manual"` гарантирует обратную совместимость — все существующие записи автоматически получат `source = "manual"` без миграции данных.

```

                                      |

                                      v
```

### 11.5. Миграции

Три последовательные миграции, все идемпотентные (`IF NOT EXISTS`):
1. CREATE TABLE `yandex_pay_partner_mapping`
2. CREATE TABLE `yandex_pay_merchant_mapping`
3. ALTER TABLE `yandex_pay_code` — добавление `source` и `merchant_id`

Связи между таблицами — на уровне приложения, не через DDL FOREIGN KEY. Это упрощает миграцию и обслуживание.

### 11.6. Изоляция данных

Все таблицы содержат `account_id`. Все запросы к БД **всегда** фильтруются по `account_id` текущей сессии. Попытка доступа к чужим данным → HTTP 403.

## Часть 12. Обратная совместимость

Это критически важная часть — мы **ничего не ломаем**.

### 12.1. Основные принципы

- **Аддитивность:** OAuth добавляет новые endpoints, UI-элементы и таблицы. Существующее не удаляется
- **Параллельность:** Оба режима (ручной и OAuth) доступны одновременно. Разные рестораны могут использовать разные способы
- **Прозрачность для плагина:** Плагин iikoFront работает одинаково. Pull-запрос возвращает токен из БД — без различия между manual и OAuth
- **Независимость контуров:** Отзыв OAuth-токена не влияет на работу плагина, если User Token уже получен и сохранён

### 12.2. Что НЕ нужно делать существующим клиентам

- Переустанавливать плагин iikoFront
- Перенастраивать iikoOffice
- Удалять старый User Token
- Перепривязывать QR-таблички
- Менять конфигурацию Api-Key

### 12.3. Feature Toggle

OAuth-функционал управляется параметром конфигурации `YandexPayOAuthEnabled`:
- По умолчанию `false` (safe roll-out)
- При `false`: блок OAuth скрыт, endpoints `/api/oauth/*` и `/api/vendor/*` → HTTP 404
- При `true`: всё работает
- Ручной ввод доступен **всегда**, независимо от toggle

### 12.4. Rollback-стратегия

Два уровня:

**Уровень 1 — Toggle off (минуты):**
- `YandexPayOAuthEnabled = false`
- OAuth скрыт, ручной режим работает, все токены и привязки сохранены

**Уровень 2 — Полный rollback (десятки минут):**
- Откат версии iikoWeb + обратные миграции БД
- Все User Token продолжают работать
- Данные маппинга утрачены, потребуется повторный онбординг

## Часть 13. Ограничения MVP

Фиксирую ключевые ограничения, чтобы не было завышенных ожиданий:

1. **Одна заявка = один ресторан.** Batch-подача нескольких мерчантов одним запросом не реализуется
2. **Максимум 10 касс на один Merchant ID.** Если больше — вторая заявка
3. **Один OAuth-аккаунт на сессию.** Для управления разными организациями — выход и повторный вход
4. **Нет фонового обновления статусов.** Обновление при открытии страницы или поллинг не чаще 1 раз в 10 минут
5. **Нет глубокой синхронизации.** Изменения в ЛК Яндекс.Пэй не отражаются в iiko автоматически
6. **QR-маппинг только на стороне iiko.** Яндекс не хранит привязку «табличка ↔ касса»
7. **Добавление новых QR-табличек — только через ЛК Яндекс.** iikoWeb их только забирает
8. **Api-Key не редактируется через OAuth.** OAuth-процесс получает только User Token
9. **Управление токенами автоматическое.** UI для ручного управления token_id не реализуется
10. **Маппинг существующих мерчантов из ЛК Яндекс не входит в MVP.** Существующие клиенты продолжают ручной ввод
11. **Касса iiko не имеет атрибута физического адреса.** Адрес доступен только на уровне RMS (ресторана)
12. **OAuth-токены хранятся в сессии** — при завершении сессии требуется повторная авторизация. Фоновое обновление не реализуется

## Часть 14. Обработка ошибок — ключевые сценарии

### Ошибки OAuth

| Ситуация | Действие | Сообщение |
|----------|---------|-----------|
| Пользователь отменил | Закрыть popup, показать кнопку «Войти» | «Авторизация отклонена. Для подключения необходимо предоставить доступ» |
| Невалидный state (CSRF) | Отклонить callback, логировать инцидент | «Ошибка безопасности. Повторите попытку» |
| Истечение authorization code | Перенаправить на повторную авторизацию | «Время авторизации истекло. Повторите попытку» |
| Popup заблокирован | Определить блокировку (`window.open = null`) | «Разрешите всплывающие окна для данного сайта» |
| Таймаут callback (5 мин) | Закрыть popup | «Время ожидания авторизации истекло» |

### Ошибки API

| HTTP | Ситуация | Действие |
|------|----------|---------|
| 400 | Невалидные данные | Подсветить поля, оставить форму открытой |
| 401 | Токен истёк | Попытка refresh → при неудаче сброс авторизации |
| 403 | Нет доступа | Предложить «Выйти» и войти под другим аккаунтом |
| 404 | Ресурс не найден | «Объект не найден. Возможно, он был удалён» |
| 500 | Сервер Яндекс недоступен | Toast, **не** повторять автоматически, логировать |

## Часть 18. Резюме для команды

### Что делаем:
Добавляем OAuth 2.0 авторизацию через Яндекс ID для автоматизации онбординга мерчантов в панели Яндекс.Пэй iikoWeb.

### Ключевые решения:
1. **Server-to-server архитектура** — все вызовы API через backend, токены не на фронтенде
2. **Feature toggle** — можно включить/выключить без деплоя кода
3. **Полная обратная совместимость** — ручной режим работает всегда
4. **Два контура авторизации** — OAuth и платёжный изолированы друг от друга
5. **Минимальные изменения в существующем** — только два поля в одной таблице
6. **Одна заявка = один ресторан** — MVP-ограничение для простоты

### Объём работы по компонентам:
- **iikoWeb Frontend:** новый блок авторизации, формы, экран статусов, интеграция с YaAuthSuggest
- **iikoWeb Backend:** 13 новых endpoints (4 OAuth + 9 proxy), 5 новых компонентов, 3 миграции БД
- **Плагин iikoFront:** минимальные изменения (получение токена через backend)

### Главный риск:
Зависимость от Яндекса — готовность Vendor Cabinet API, уточнение scopes, время обработки заявок.
