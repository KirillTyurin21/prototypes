> **Формат:** речь системного аналитика для встречи с командой разработки

> **Прототип:** https://view21.ru/prototype/aurora  
> **Самодостаточный документ** — все детали, поля, методы и параметры описаны здесь, без необходимости обращения к спецификации

## 1. Что мы делаем и зачем

Коллеги, сегодня расскажу про проект WB Pay — интеграцию с Wildberries Pay для наших POS-терминалов.

**Бизнес-контекст.** Wildberries запускает свой платёжный кошелёк — WB-кошелек. Это оплата через QR-код из приложения Wildberries. Они хотят выйти за рамки маркетплейса и охватить HoReCa — рестораны, кафе, фастфуд. У нас около 80 000 касс на iikoFront — это большая база для WB.

**Что конкретно мы делаем.** Мы создаём две вещи:

1. **Плагин для iikoFront** — .NET-плагин, который реализует интерфейс `IExternalPaymentProcessor` (iikoFront API V9). Это стандартный способ подключения внешних платёжных систем. Плагин добавляет новый тип оплаты «WB-кошелек» на кассу.

2. **Страницу администрирования в iikoWeb** — раздел «Интеграции → WB Pay», где администратор управляет credentials (ключи доступа) для ресторанов и терминалов.

**Как это выглядит для кассира.** Гость подходит к кассе. Кассир пробивает заказ, нажимает оплату, выбирает тип оплаты «WB-кошелек». Сканирует QR-код с телефона гостя. Гость подтверждает оплату в приложении WB. Деньги списаны, чек напечатан, заказ закрыт. Всё — кассир работает в привычном интерфейсе iikoFront, никаких новых окон и приложений не нужно.

## 2. Архитектура решения — общая картина

Теперь по архитектуре. У нас три ключевых компонента и две линии связи:

```
┌─────────────────┐          HTTPS/REST           ┌──────────────┐
│  Плагин iikoFront│ ──────────────────────────── │  WB Pay API  │
│  (.NET, V9 SDK) │     api.wbpay.ru              │  (Wildberries)│
└────────┬────────┘                                └──────────────┘
         │
         │  iikoTransport (Cloud)
         │  ← push credentials (WBPay/CredentialsUpdated)
         │  → pull credentials (при запуске плагина)
         │
┌────────┴────────┐
│  iikoWeb         │
│  (админ-панель)  │
└─────────────────┘
```

**Линия 1: Плагин → WB Pay API.** Все платёжные операции — оплата, возврат — идут напрямую из плагина в REST API WB Pay по HTTPS. Это основной операционный канал. Базовый URL: `https://api.wbpay.ru` (продакшен) или `https://sandbox.wbpay.ru` (sandbox).

**Линия 2: iikoWeb → Плагин через Transport.** Управление настройками. Администратор сохраняет credentials в iikoWeb, они автоматически доставляются на терминалы через iikoTransport (облачный канал). Это канал конфигурации — без него платежи не начнутся, потому что плагину нужны ключи.

## 3. WB Pay API — как устроено взаимодействие

### 3.1. Эндпоинты

WB Pay предоставляет REST API. JSON, HTTPS, синхронные вызовы. У них **6 эндпоинтов** — три для оплаты, три для возврата:

| # | Метод | Эндпоинт | Назначение |
|:-:|-------|----------|------------|
| 1 | POST | `/api/v1/orders/offline/register` | Создать заказ → получить `order_id` |
| 2 | POST | `/api/v1/orders/do` | Подтвердить оплату |
| 3 | GET | `/api/v1/orders/{order_id}/status` | Опрос статуса оплаты (polling) |
| 4 | POST | `/api/v1/refunds/register` | Создать возврат → получить `refund_id` |
| 5 | POST | `/api/v1/refunds/do` | Подтвердить возврат |
| 6 | GET | `/api/v1/refunds/{refund_id}/status` | Опрос статуса возврата (polling) |

Обратите внимание на `offline` в пути `/orders/offline/register` — это означает оффлайн-режим оплаты WB (через POS-терминал, а не онлайн-магазин). Для возвратов `/refunds/register` слова `offline` нет.

Нет webhook'ов. WB Pay не отправляет нам уведомления о смене статуса. Мы используем **polling-модель**: после вызова `/do` плагин периодически опрашивает `/status`, пока не получит финальный статус (`succeeded` или `failed`).

### 3.2. Трёхшаговая модель

Каждая операция (и оплата, и возврат) проходит три шага:

1. **Register** — регистрируем намерение. Получаем идентификатор (`order_id` или `refund_id`). У идентификатора **TTL = 2 минуты** — за это время нужно вызвать `/do`, иначе идентификатор протухнет.
2. **Do** — подтверждаем операцию. Это точка невозврата. Метод **идемпотентен** — повторный вызов с тем же ID безопасен, двойного списания не будет.
3. **Status (polling)** — опрашиваем статус каждые 2–3 секунды, пока не получим финальный ответ (`succeeded` или `failed`).

### 3.3. HTTP-заголовки — что передаётся в каждом запросе

| Заголовок | Формат | Где используется | Откуда значение |
|-----------|--------|:----------------:|-----------------|
| `Authorization` | `Bearer {JWT}` | **Все 6** эндпоинтов | `config.xml → JwtToken` |
| `X-Signature` | Base64(ED25519_sign(body)) | Только `/register` (эндпоинты #1 и #4) | Подпись тела запроса ключом из `private.pem` |
| `X-Wbpay-Id` | `{order_id}` | Эндпоинты #2, #3, #4, #5, #6 | `order_id` из ответа `/register` (оригинальный) |
| `X-Request-Country` | ISO 3166-1 alpha-2, напр. `RU` | Только `/orders/offline/register` (#1) | `config.xml → Country` |
| `X-Request-Region` | ISO 3166-2, напр. `RU-MOW` | Только `/orders/offline/register` (#1) | `config.xml → Region` |
| `Content-Type` | `application/json` | Все POST-запросы | Константа |

**Про X-Signature (ED25519-подпись).** Применяется только к двум `/register` эндпоинтам. Алгоритм:
1. Берём сырое тело запроса (JSON, UTF-8 байты)
2. Подписываем приватным ключом ED25519 из файла `private.pem`
3. Результат кодируем в Base64
4. Передаём в заголовке: `X-Signature: {base64_подпись}`

**Про X-Wbpay-Id.** Важный нюанс для возвратов: в заголовке `X-Wbpay-Id` всегда передаётся `order_id` **оригинальной оплаты**, а не `refund_id`. Это относится ко всем эндпоинтам возврата (#4, #5, #6).

### 3.4. Авторизация

Каждый запрос к WB Pay содержит два уровня защиты:

1. **JWT Bearer Token** — в заголовке `Authorization: Bearer {JWT}`. Передаётся **в каждом** запросе. TTL токена — **1 год**. Механизм автоматического обновления отсутствует — обновление вручную через iikoWeb.
2. **ED25519-подпись** — в заголовке `X-Signature`. Передаётся **только на эндпоинтах `/register`**. Подпись вычисляется по телу запроса с использованием приватного ключа `private.pem`.

Плюс в запросах передаётся `terminal_id` (в теле) — идентификатор терминала в системе WB Pay. Это привязка «какая конкретно касса делает запрос».

## 4. Оплата — детальный сценарий с полями

### 4.1. Happy path — по шагам

**Шаг 1. Кассир выбирает тип оплаты.**
Кассир нажимает «WB-кошелек» в экране оплаты iikoFront. SDK вызывает метод `Pay()` нашего плагина.

Что мы получаем в `Pay()`:

| Параметр SDK | Тип | Что берём из него |
|-------------|-----|-------------------|
| `paymentItem.Sum` | decimal | Сумма оплаты в рублях (потом × 100 → копейки для API) |
| `order.Id` | Guid | GUID заказа → передаём как `invoice_id` в API |
| `order.Items` | IOrderProductItem[] | Корзина товаров → массив `positions[]` для API |

---

**Шаг 2. Диалог сканирования QR.**
Плагин вызывает `ShowExtendedInputDialog()`:
- Заголовок: `"Оплата WB-кошельком"`
- Сообщение: `"Отсканируйте QR-код из приложения Wildberries"`
- `enableBarcode = true` (для HID-сканера)
- `enableKeyboard = true` (для ручного ввода при тестировании)

Кассир сканирует QR с телефона гостя (гость открыл его в приложении WB). Из результата диалога берём `BarcodeValue` — строка QR-кода, она же `qr_code` для API.

Если кассир нажал «Отмена» → `Accepted = false` → бросаем `PaymentActionCancelledException`.

> TTL QR-кода гостя ≈ **5 минут**. Если просрочен — гость должен обновить QR в приложении WB.

---

**Шаг 3. POST /api/v1/orders/offline/register**

Плагин отправляет запрос на регистрацию заказа.

**Заголовки:**

| Заголовок | Значение | Источник |
|-----------|----------|----------|
| `Authorization` | `Bearer {JWT}` | `config.xml → JwtToken` |
| `X-Signature` | Base64(ED25519_sign(body)) | Подпись тела ключом из `private.pem` |
| `X-Request-Country` | `RU` | `config.xml → Country` |
| `X-Request-Region` | `RU-MOW` | `config.xml → Region` |

**Тело запроса:**

```json
{
  "amount": 5000,
  "currency_code": 643,
  "qr_code": "строка из QR-кода гостя",
  "terminal_id": "WB_TERMINAL_001",
  "created_at": 1772627112,
  "invoice_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "positions": [
    { "count": 2, "name": "Капучино", "price": 2500 },
    { "count": 1, "name": "Круассан", "price": 1800 }
  ]
}
```

| Поле | Тип | Описание | Откуда значение |
|------|-----|----------|-----------------|
| `amount` | int64 | Сумма в **копейках** (50 руб = 5000) | `paymentItem.Sum × 100` |
| `currency_code` | int64 | Код валюты. Всегда **643** (RUB) | Константа |
| `qr_code` | string | Строка QR-кода гостя | `ShowExtendedInputDialog().BarcodeValue` |
| `terminal_id` | string | ID терминала в системе WB Pay | `config.xml → TerminalId` |
| `created_at` | int64 | Unix timestamp UTC (время инициации оплаты) | `DateTime.UtcNow` → Unix timestamp |
| `invoice_id` | string | GUID заказа в iiko (макс. 2048 символов) | `order.Id` (IOrder.Id) |
| `positions` | Position[] | Корзина товаров (макс. **1000** позиций) | `order.Items` → преобразование |
| `positions[].count` | int64 | Количество штук (> 0) | `IOrderProductItem.Amount` |
| `positions[].name` | string | Название товара (макс. 2048 символов) | `IOrderProductItem.Product.Name` |
| `positions[].price` | int64 | Цена за единицу в **копейках** (> 0) | `IOrderProductItem.Price × 100` |

**Ответ (HTTP 200):**

```json
{
  "error_code": "ERR_NONE",
  "error_description": "",
  "data": {
    "order_id": "b2c3d4e5-f6g7-8901-bcde-f12345678901"
  }
}
```

| Поле ответа | Тип | Что с ним делаем |
|-------------|-----|------------------|
| `error_code` | string | `"ERR_NONE"` = успех. Иначе — ошибка → показать `error_description` кассиру |
| `error_description` | string | Текст ошибки (пусто при успехе) |
| `data.order_id` | string (UUID) | **Главное значение.** Сохраняем через `SetRollbackData()` (для будущих возвратов) и используем в следующих шагах |

> **TTL `order_id` = 2 минуты.** За это время нужно вызвать `/do`.

---

**Шаг 4 (между register и do). Сохранение order_id для возвратов.**

Сразу после получения `order_id` вызываем SDK-метод:

```csharp
context.SetRollbackData(new { order_id = orderIdFromResponse });
```

Это критически важно: `order_id` привязывается к транзакции и будет доступен при будущем вызове `ReturnPayment()` или `EmergencyCancelPayment()` через `GetRollbackData()`.

---

**Шаг 5. POST /api/v1/orders/do**

Подтверждаем оплату.

**Заголовки:**

| Заголовок | Значение | Источник |
|-----------|----------|----------|
| `Authorization` | `Bearer {JWT}` | `config.xml → JwtToken` |
| `X-Wbpay-Id` | `{order_id}` | `order_id` из ответа шага 3 |

> Обратите внимание: `X-Signature` здесь **не передаётся** — подпись только на `/register`.

**Тело запроса:**

```json
{
  "order_id": "b2c3d4e5-f6g7-8901-bcde-f12345678901"
}
```

| Поле | Тип | Описание | Откуда |
|------|-----|----------|--------|
| `order_id` | string (UUID) | ID оплаты из предыдущего шага | Ответ `POST /orders/offline/register` → `data.order_id` |

**Ответ:**

```json
{
  "error_code": "ERR_NONE",
  "error_description": ""
}
```

Ответ содержит только `error_code` и `error_description` — данных (`data`) нет. Если `error_code != "ERR_NONE"` → ошибка.

> **Идемпотентность:** повторный вызов с тем же `order_id` безопасен — двойного списания не будет.

---

**Шаг 6. Ожидание подтверждения гостя.**

После `/do` WB Pay ждёт, пока гость подтвердит оплату в приложении. Плагин показывает кассиру диалог ожидания: «Попросите гостя подтвердить оплату в приложении WB». Прогресс-бар, сумма, кнопка отмены.

---

**Шаг 7. Polling: GET /api/v1/orders/{order_id}/status**

Плагин циклически опрашивает статус.

**Заголовки:**

| Заголовок | Значение | Источник |
|-----------|----------|----------|
| `Authorization` | `Bearer {JWT}` | `config.xml → JwtToken` |
| `X-Wbpay-Id` | `{order_id}` | Тот же `order_id` |

**URL:** `/api/v1/orders/{order_id}/status` — `order_id` подставляется прямо в путь.

**Тела запроса нет** (GET-запрос).

**Ответ:**

```json
{
  "error_code": "ERR_NONE",
  "error_description": "",
  "data": {
    "status": "succeeded",
    "fail_reason_code": "ERR_NONE",
    "fail_reason_description": ""
  }
}
```

| Поле ответа | Тип | Описание |
|-------------|-----|----------|
| `data.status` | string | Текущий статус операции |
| `data.fail_reason_code` | string | Код причины отказа (при `status = failed`) |
| `data.fail_reason_description` | string | Текст причины на русском (при `status = failed`) |

**Маппинг статусов:**

| Статус | Финальный? | Действие плагина |
|--------|:----------:|------------------|
| `created` | Нет | Продолжить polling |
| `pending` | Нет | Продолжить polling |
| `succeeded` | **Да** | Завершить `Pay()` без исключений → оплата успешна |
| `failed` | **Да** | Бросить `PaymentActionFailedException` + показать `fail_reason_description` |

**Параметры polling:**
- **Интервал:** 2–3 секунды между запросами (из `config.xml → PollingIntervalMs`, по умолчанию 3000 мс)
- **Таймаут:** `config.xml → PollingTimeoutSec`, по умолчанию 120 секунд
- **Условие выхода:** получен `status = succeeded` или `status = failed`

---

**Шаг 8. Успех.**

При `status = succeeded`: показываем кассиру диалог «Оплата прошла успешно» + сумму + `order_id`. Кассир нажимает ОК, iikoFront печатает чек и закрывает заказ.

---

### 4.2. Полный реестр ошибок оплаты

При `status = failed` — в `data.fail_reason_code` приходит код причины. Вот полный маппинг кодов на сообщения кассиру:

| `fail_reason_code` | Причина | Сообщение кассиру |
|--------------------|---------|-------------------|
| `EXPIRED_QR_CODE` | QR-код просрочен (TTL ≈ 5 мин) | «QR-код просрочен. Попросите гостя обновить QR в приложении WB» |
| `INVALID_QR_CODE` | Невалидный формат QR | «Невалидный QR-код. Попросите гостя показать новый QR» |
| `DUPLICATE_QR_CODE` | QR уже был использован | «QR-код уже использован. Попросите гостя обновить QR в приложении WB» |
| `NOT_ENOUGH_MONEY` | Недостаточно средств | «Недостаточно средств в WB-кошельке» |
| `LIMIT_EXCEEDED` | Сумма превышает лимит | «Сумма превышает лимит WB-кошелька» |
| `NO_AVAILABLE_PAYMENT_METHODS` | У гостя не привязана карта | «У гостя не привязан способ оплаты в приложении WB» |
| `CONFIRMATION_TIME_EXPIRED` | Гость не подтвердил вовремя | «Гость не подтвердил оплату вовремя. Повторите» |
| `ORDER_EXPIRED` | TTL `order_id` истёк (2 мин) | «Время операции истекло. Повторите оплату» |
| `NOT_FOUND` | `order_id` не найден | «Время операции истекло. Повторите оплату» |
| `UNABLE_TO_PROCESS` | Общий отказ WB | «Оплата отклонена. Попробуйте позже или выберите другой способ оплаты» |
| *Неизвестный код* | — | Показать `fail_reason_description` как есть |

**HTTP-ошибки (не связанные с бизнес-логикой):**

| Ситуация | Сообщение кассиру |
|----------|-------------------|
| Нет связи (timeout, DNS, connection refused) | «Нет связи с сервером WB Pay. Проверьте подключение к интернету и повторите попытку» |
| HTTP 403 (JWT невалиден) | «Ошибка авторизации WB Pay. Обратитесь к администратору для проверки настроек плагина» |
| Polling таймаут (120 сек) | «Время ожидания оплаты истекло. Повторите попытку» |

Все сообщения — на русском, понятном кассиру языке. Никаких технических деталей на экране кассы. Плагин бросает `PaymentActionFailedException(сообщение_кассиру)`.

## 5. Возврат — детальный сценарий с полями

### 5.1. Happy path — по шагам

Возврат инициируется стандартно через iikoFront — SDK вызывает `ReturnPayment()` нашего плагина.

**Шаг 1. Получение `order_id` оригинальной оплаты.**

```csharp
var rollbackData = context.GetRollbackData<dynamic>();
string originalOrderId = rollbackData.order_id;
```

`GetRollbackData()` возвращает объект, который мы сохранили при оплате через `SetRollbackData()`. Там лежит `order_id` — идентификатор оригинальной оплаты в WB Pay. Без него возврат невозможен.

Если `RollbackData` отсутствует → показать кассиру: «Данные исходной оплаты не найдены. Возврат невозможен».

---

**Шаг 2. POST /api/v1/refunds/register**

**Заголовки:**

| Заголовок | Значение | Источник |
|-----------|----------|----------|
| `Authorization` | `Bearer {JWT}` | `config.xml → JwtToken` |
| `X-Signature` | Base64(ED25519_sign(body)) | Подпись тела ключом из `private.pem` |
| `X-Wbpay-Id` | `{order_id}` | `order_id` **оригинальной оплаты** (из `GetRollbackData()`) |

**Тело запроса:**

```json
{
  "amount": 5000,
  "currency_code": 643,
  "order_id": "b2c3d4e5-f6g7-8901-bcde-f12345678901",
  "terminal_id": "WB_TERMINAL_001",
  "created_at": 1772627200,
  "invoice_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "positions": [
    { "count": 2, "name": "Капучино", "price": 2500 },
    { "count": 1, "name": "Круассан", "price": 1800 }
  ]
}
```

| Поле | Тип | Описание | Откуда значение |
|------|-----|----------|-----------------|
| `amount` | int64 | Сумма возврата в **копейках** (MVP: только полный возврат = вся сумма оригинальной оплаты) | `paymentItem.Sum × 100` |
| `currency_code` | int64 | Всегда **643** (RUB) | Константа |
| `order_id` | string (UUID) | ID **оригинальной** оплаты (по нему WB привязывает возврат) | `GetRollbackData().order_id` |
| `terminal_id` | string | ID терминала в WB Pay | `config.xml → TerminalId` |
| `created_at` | int64 | Unix timestamp UTC момента возврата | `DateTime.UtcNow` → Unix timestamp |
| `invoice_id` | string | GUID заказа в iiko | `order.Id` (IOrder.Id) |
| `positions` | Position[] | Те же позиции, что в оригинальной оплате | `order.Items` → преобразование |

**Ответ (HTTP 200):**

```json
{
  "error_code": "ERR_NONE",
  "error_description": "",
  "data": {
    "refund_id": "c3d4e5f6-g7h8-9012-cdef-g23456789012"
  }
}
```

| Поле ответа | Что с ним делаем |
|-------------|------------------|
| `data.refund_id` | Используем в следующих шагах (`/do` и `/status`). TTL = 2 минуты |

---

**Шаг 3. POST /api/v1/refunds/do**

**Заголовки:**

| Заголовок | Значение | Источник |
|-----------|----------|----------|
| `Authorization` | `Bearer {JWT}` | `config.xml → JwtToken` |
| `X-Wbpay-Id` | `{order_id}` | `order_id` **оригинальной оплаты** (не `refund_id`!) |

**Тело запроса:**

```json
{
  "refund_id": "c3d4e5f6-g7h8-9012-cdef-g23456789012"
}
```

| Поле | Тип | Описание | Откуда |
|------|-----|----------|--------|
| `refund_id` | string (UUID) | ID возврата из предыдущего шага | Ответ `/refunds/register` → `data.refund_id` |

> **Идемпотентность:** повторный вызов с тем же `refund_id` безопасен.

---

**Шаг 4. Polling: GET /api/v1/refunds/{refund_id}/status**

**URL:** `/api/v1/refunds/{refund_id}/status` — `refund_id` подставляется в путь.

**Заголовки:**

| Заголовок | Значение | Источник |
|-----------|----------|----------|
| `Authorization` | `Bearer {JWT}` | `config.xml → JwtToken` |
| `X-Wbpay-Id` | `{order_id}` | `order_id` **оригинальной оплаты** (не `refund_id`!) |

**Ответ:** та же структура, что для оплаты — `data.status`, `data.fail_reason_code`, `data.fail_reason_description`.

**Маппинг статусов:** идентичен оплате (`created`/`pending` → polling, `succeeded` → OK, `failed` → ошибка).

---

**Шаг 5. Успех.**

При `status = succeeded`: кассиру — «Возврат выполнен успешно».

### 5.2. Ошибки возврата

| `fail_reason_code` | Причина | Сообщение кассиру |
|--------------------|---------|-------------------|
| `REFUND_NOT_POSSIBLE` | Возврат по этой оплате невозможен | «Возврат по данной оплате невозможен. Обратитесь в поддержку WB Pay» |
| `REFUND_EXPIRED` | Превышен лимит времени возврата | «Время операции возврата истекло. Повторите» |
| `NOT_FOUND` | `refund_id` не найден (TTL 2 мин) | «Время операции возврата истекло. Повторите возврат» |
| `UNABLE_TO_PROCESS` | Общая ошибка | «Возврат отклонён. Попробуйте позже» |
| `SYSTEM_ERROR` | Внутренняя ошибка WB | «Системная ошибка WB Pay. Повторите позже» |

### 5.3. Ограничение MVP

В рамках MVP — **только полные возвраты** (`amount` = полная сумма оригинальной оплаты). Частичные возвраты — на MVP+. API WB Pay уже поддерживает частичные, нам нужно только доработать плагин.

## 6. FISCAL_ERROR — критический сценарий

Это самый важный edge-case, и его нужно понимать всей команде.

### 6.1. Что такое FISCAL_ERROR

В iikoFront есть настройка `FiscalChequeBeforePaymentEnabled`. Она определяет, когда печатается фискальный чек — **до** или **после** оплаты.

Если фискальный чек печатается **после** оплаты (`FiscalChequeBeforePaymentEnabled = false`) и при печати возникает ошибка (бумага кончилась, принтер сломался, ФР заблокирован), iikoFront генерирует событие **FISCAL_ERROR**. В этот момент деньги уже списаны с кошелька гостя, но фискальный чек не напечатан — нарушение ФЗ-54.

### 6.2. Что делает плагин

iikoFront вызывает метод `EmergencyCancelPayment()` нашего плагина (сигнатура идентична `ReturnPayment()`). Плагин **обязан** выполнить полный возврат средств через WB Pay API:

1. Получить `order_id` из `GetRollbackData()` (тот же SDK-метод, что и при обычном возврате)
2. `POST /api/v1/refunds/register` — регистрируем экстренный возврат (тело и заголовки как в обычном возврате)
3. `POST /api/v1/refunds/do` — подтверждаем возврат
4. Polling `GET /api/v1/refunds/{refund_id}/status` — ждём финального статуса
5. Если `succeeded` → кассиру: «Возврат выполнен. Повторите оплату после устранения проблемы с чеком»

### 6.3. Если экстренный возврат не прошёл

Это **критическая** ситуация. Деньги списаны, чек не напечатан, возврат не удался.

| Ситуация | Действие плагина |
|----------|------------------|
| WB Pay вернул `status = failed` | Показать `fail_reason_description` + «Обратитесь в поддержку WB Pay для ручного возврата» |
| Polling таймаут | «Экстренный возврат не завершён вовремя. Обратитесь в поддержку WB Pay» |
| `GetRollbackData()` не вернул данных | «Данные исходной оплаты не найдены. Обратитесь в поддержку WB Pay» |

Логируем на уровне **Error** с полной информацией: `order_id`, `refund_id`, `fail_reason_code`, `fail_reason_description`.

### 6.4. Рекомендация по конфигурации

Мы **настоятельно** рекомендуем `FiscalChequeBeforePaymentEnabled = true` — печать чека **до** списания денег. В этом случае FISCAL_ERROR **невозможен**: если чек не печатается, оплата просто не проводится. Соответствует п. 5.11 ФЗ-54.

## 7. SDK-методы плагина — полный справочник

### 7.1. Интерфейс `IExternalPaymentProcessor`

Плагин реализует три метода этого интерфейса:

#### `Pay()`

```csharp
void Pay(
    IPaymentItem paymentItem,           // .Sum = сумма в рублях
    IOrder order,                        // .Id = GUID заказа, .Items = товары
    IOperationService operationService,  // Для SetRollbackData
    IPluginOperationContext context       // Контекст операции
);
```

**Что делает:** Полный цикл оплаты (шаги 1–8 из раздела 4.1).

**При успехе:** возвращает без исключений → iikoFront считает оплату проведённой.
**При отмене кассиром:** бросает `PaymentActionCancelledException`.
**При ошибке WB Pay:** бросает `PaymentActionFailedException(сообщение_кассиру)`.

#### `ReturnPayment()`

```csharp
void ReturnPayment(
    IPaymentItem paymentItem,
    IOrder order,
    IOperationService operationService,
    IPluginOperationContext context
);
```

**Что делает:** Полный цикл возврата (шаги 1–5 из раздела 5.1).

#### `EmergencyCancelPayment()`

Сигнатура идентична `ReturnPayment()`. Вызывается iikoFront при FISCAL_ERROR. Выполняет экстренный возврат (раздел 6.2).

### 7.2. Вспомогательные SDK-методы

#### `RegisterPaymentSystem()` — регистрация типа оплаты

```csharp
operationService.RegisterPaymentSystem(
    "WildberriesPay",    // paymentSystemKey — уникальный ID в системе
    "WB-кошелек",        // paymentSystemName — отображается кассиру
    canProcessPaymentReturnWithoutOrder: false  // WB Pay требует order_id для возврата
);
```

**Когда вызывается:**
- При запуске плагина — если config.xml содержит валидные credentials
- После успешной QR-проливки — сразу после записи credentials
- После получения credentials через Transport push

**Эффект:** «WB-кошелек» появляется в списке типов оплаты на экране кассы.

> **Иконка:** программно через SDK установить невозможно — `RegisterPaymentSystem()` принимает только `paymentSystemKey` и `paymentSystemName`. Иконка задаётся DevOps через Groovy-скрипт при создании плагинного типа оплаты.

#### `SetRollbackData()` / `GetRollbackData()`

```csharp
// При оплате — сохраняем order_id:
context.SetRollbackData(new { order_id = orderIdFromResponse });

// При возврате — извлекаем order_id:
var data = context.GetRollbackData<dynamic>();
string orderId = data.order_id;
```

Данные привязаны к конкретной транзакции оплаты. Используются для связи оплата → возврат.

#### `ShowExtendedInputDialog()` — диалог со сканером

```csharp
var result = viewManager.ShowExtendedInputDialog(
    title: "Оплата WB-кошельком",
    message: "Отсканируйте QR-код из приложения Wildberries",
    enableBarcode: true,    // HID-сканер
    enableKeyboard: true    // Ручной ввод

// result.Accepted — true/false (ОК/Отмена)
// result.BarcodeValue — строка QR-кода
```

#### `AddButton()` — кнопка в меню «Дополнения»

```csharp
integrationService.AddButton(
    "WB Pay Setup",                              // buttonName
    "Настройка WB Pay",                          // caption (текст на кнопке)
    PluginButtonPosition.AdvancedOperations,      // позиция: меню «Дополнения»
    handler                                       // обработчик нажатия

```

Кнопка регистрируется **всегда** при старте плагина, независимо от наличия credentials. Через неё администратор может запустить QR-проливку (канал B).

### 7.3. Лицензирование — ModuleId

```csharp
[PluginLicenseModuleId(XXXXXXXXX18)]  // Заканчивается на "18" (PaymentAPI)
public class WildberriesPayPlugin : IExternalPaymentProcessor { }
```

- ModuleId создаётся командой ВА через Pyrus-форму
- **Обязательно** совпадает в двух местах: `Manifest.xml` и атрибут `[PluginLicenseModuleId]`

- Суффикс `18` = региональный модуль → бесплатное развёртывание на всех ~80 000 касс

## 8. Онбординг — как credentials попадают на терминал

Три канала доставки credentials (JWT-токен, приватный ключ, terminal_id) на кассовый терминал.

### 8.1. Канал A: iikoWeb + Transport (рекомендуемый)

Для сетевых ресторанов с подключением к iikoCloud.

**Процесс:**

1. Администратор → iikoWeb → «Интеграции → WB Pay»
2. Выбирает ресторан → вводит `terminal_id`, JWT, загружает `private.pem` → «Сохранить»
3. iikoWeb валидирует данные, генерирует `revision` (timestamp UTC мс), отправляет push

**Push-сообщение: `WBPay/CredentialsUpdated`**

```json
{
  "terminal_id": "WB_TERMINAL_001",
  "jwt_token": "eyJhbGciOiJFZERTQSIsInR5cCI6IkpXVCJ9...",
  "private_key_pem": "-----BEGIN PRIVATE KEY-----\nMC4CAQAwBQYDK2VwBCIEIGpK...\n-----END PRIVATE KEY-----",
  "revision": 1714123456789
}
```

| Поле payload | Тип | Описание | Куда записывается на терминале |
|--------------|-----|----------|-------------------------------|
| `terminal_id` | string | ID точки приёма оплат в WB Pay | `config.xml → TerminalId` |
| `jwt_token` | string | Bearer JWT (~1000-1500 байт) | `config.xml → JwtToken` |
| `private_key_pem` | string | ED25519 ключ в PEM-формате | Файл `private.pem` рядом с DLL плагина |
| `revision` | long | Timestamp UTC в миллисекундах (монотонно возрастает) | `config.xml → CredentialsRevision` |

**Как плагин обрабатывает push:**

| Действие | Условие |
|----------|---------|
| Применить credentials | `revision_входящая > revision_локальная` (из `config.xml → CredentialsRevision`) |
| Игнорировать push | `revision_входящая ≤ revision_локальная` → защита от устаревших сообщений |
| Удалить credentials | Все поля пустые (`terminal_id = ""`, `jwt_token = ""`) + `revision` > локальной → деактивировать «WB-кошелек» |

**Pull при запуске** — плагин при каждом старте отправляет pull-запрос к iikoWeb через Transport:

```json
{
  "terminalName": "Касса 1",
  "current_revision": 1714123456789
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `terminalName` | string | Имя кассы в iikoFront (для отображения в дереве iikoWeb) |
| `current_revision` | long | Текущая `CredentialsRevision` из `config.xml` (0 если нет credentials) |

**Ответы на pull:**

| HTTP-код | Что значит | Действие плагина |
|:--------:|-----------|------------------|
| 200 | Новые credentials (revision сервера > current_revision) | Валидировать → записать в config.xml + private.pem → зарегистрировать «WB-кошелек» |
| 204 | Credentials актуальны (или не настроены в iikoWeb) | Если config.xml заполнен → использовать локальные. Если пуст → канал B (QR-проливка) |
| Ошибка сети | Timeout, connection refused | Использовать локальный config.xml. Повторить pull при следующем запуске |

### 8.2. Канал B: QR-проливка (fallback)

Для автономных точек без стабильного подключения к iikoCloud.

**Генерация QR в iikoWeb:**
Администратор → iikoWeb → выбирает ресторан с сохранёнными credentials → «Сгенерировать QR». iikoWeb формирует JSON с credentials, кодирует в Base64, генерирует QR-код.

**Формат QR-кода (JSON → Base64):**

```json
{
  "terminal_id": "WB_TERMINAL_001",
  "jwt": "eyJhbGciOiJFZERTQSIsInR5cCI6IkpXVCJ9...",
  "private_key": "-----BEGIN PRIVATE KEY-----\nMC4CAQAwBQYDK2VwBCIEIGpK...\n-----END PRIVATE KEY-----",
  "revision": 1714123456789
}
```

| Поле QR | Тип | Описание |
|---------|-----|----------|
| `terminal_id` | string | ID точки приёма оплат |
| `jwt` | string | Bearer JWT (макс ~1500 байт) |
| `private_key` | string | ED25519 ключ в PEM (макс ~128 байт) |
| `revision` | long | Серверная revision из iikoWeb |

> Обратите внимание: имена полей в QR-коде отличаются от Transport push! В QR: `jwt` и `private_key`. В Transport: `jwt_token` и `private_key_pem`.

Размер Base64-строки: ~700-1700 байт → QR версии 15-25.

**На кассе — процесс активации:**

1. Кассир/техник открывает меню «Дополнения» → «Настройка WB Pay»
2. Открывается диалог `ShowExtendedInputDialog()`: «Отсканируйте QR-код настройки, сгенерированный в iikoWeb (Интеграции → WB Pay)»
3. Сканирует QR 2D-сканером (HID-режим) или вставляет Base64 вручную
4. Плагин декодирует Base64 → парсит JSON → валидирует (все поля непустые, формат корректен)
5. Записывает: `TerminalId` в config.xml, `JwtToken` в config.xml, `private.pem` как файл, `CredentialsRevision = revision` из QR
6. Вызывает `RegisterPaymentSystem()` → «WB-кошелек» активирован

**Ошибки обработки QR:**

| Ситуация | Сообщение |
|----------|-----------|
| Невалидный Base64 | «Невалидный QR-код настройки. Убедитесь, что сканируете QR-код из iikoWeb (Интеграции → WB Pay)» |
| JSON неполный (отсутствует поле) | «QR-код не содержит все необходимые данные. Сгенерируйте новый QR в iikoWeb» |
| Ошибка записи файла (нет прав) | «Не удалось сохранить настройки. Проверьте права доступа к папке плагина» |

### 8.3. Канал C: Ручная настройка (аварийный)

Вручную: отредактировать `config.xml` (вписать `TerminalId`, `JwtToken`), скопировать файл `private.pem` в папку плагина. `CredentialsRevision = 0` (без revision → будет перезаписан следующим push).

Используется **только** если каналы A и B недоступны.

### 8.4. Последовательность инициализации при запуске

Полное дерево решений при старте плагина:

```
Запуск плагина
│
├─ 1. Зарегистрировать Cloud-обработчик WBPay/CredentialsUpdated
│     (для приёма push в любой момент работы)
│
├─ 2. Зарегистрировать кнопку «Настройка WB Pay» в меню «Дополнения»
│     (ВСЕГДА, даже без credentials)
│
├─ 3. Прочитать config.xml
│     │
│     ├─ config.xml не найден → Error в лог → НЕ регистрируем → СТОП
│     ├─ XML невалиден → Error в лог → НЕ регистрируем → СТОП
│     │
│     └─ config.xml прочитан
│        │
│        ├─ TerminalId пуст → НЕ регистрируем (но pull пробуем)
│        ├─ JwtToken пуст → НЕ регистрируем (но pull пробуем)
│        │
│        └─ Credentials заполнены → продолжаем
│
├─ 4. Pull-запрос к iikoWeb (текущая revision)
│     │
│     ├─ HTTP 200 + revision > локальной → Обновить config.xml + private.pem
│     ├─ HTTP 204 → Использовать локальные
│     ├─ Ошибка сети → Использовать локальные
│     │
│     └─ Если config.xml всё ещё пуст → показать диалог QR-проливки (канал B)
│
├─ 5. Валидировать private.pem (файл существует и читаем)
│     │
│     ├─ Файл не найден → Error → НЕ регистрируем → СТОП
│     └─ Файл OK → продолжаем
│
└─ 6. RegisterPaymentSystem("WildberriesPay", "WB-кошелек")
      → «WB-кошелек» доступна кассиру ✓
```

## 9. Страница iikoWeb — панель администрирования

### 9.1. Расположение и доступ

Раздел «Интеграции → WB Pay» в iikoWeb. Доступен пользователям с ролью **«Администратор»**.

### 9.2. Layout — двухпанельный интерфейс

**Левая панель** — дерево организаций:

```
Ресторанная группа «Восток»        (организация)
├── Пицца Маргарита                 ● зелёный (настроен)
│   ├── Касса 1                     ✓ credentials применены
│   └── Касса 2                     ✓ credentials применены
├── Суши Сакура                     ○ серый (не настроен)
│   └── Касса 1
└── Бургер Хаус                     ○ серый (не настроен)
    └── Касса 1
```

Цветовые индикаторы: зелёный = настроен, серый = не настроен, красный = ошибка.

**Правая панель** — детали выбранного элемента:

**При выборе ресторана — форма credentials:**

| Поле | Тип контрола | Описание | Валидация |
|------|-------------|----------|-----------|
| `terminal_id` | Текстовое поле | ID точки приёма оплат в WB Pay | Не пустой |
| `JWT Bearer Token` | Textarea (моноширинный) | JWT-токен авторизации | Формат: 3 части через точку (`header.payload.signature`) |
| `private.pem` | Textarea + загрузка файла | ED25519 приватный ключ | Начинается с `-----BEGIN PRIVATE KEY-----` |

Кнопки:
- **Сохранить** → валидация → если OK → генерация revision → push на все терминалы ресторана через Transport
- **Сгенерировать QR** → активна только если credentials сохранены → модалка с QR
- **Удалить** (красная) → подтверждение → push пустых credentials → деактивация на терминалах

**При выборе терминала — read-only информация:**

| Поле | Описание |
|------|----------|
| ID терминала | Из последнего pull-запроса терминала |
| Ресторан | К какому ресторану принадлежит |
| Последний запрос | Дата/время последнего pull |
| Источник конфигурации | `transport` / `qr` / `manual` / `нет` |
| Credentials применены | Да / Нет |

### 9.3. Валидация credentials при сохранении

| Проверка | Условие ошибки | Реакция |
|----------|---------------|---------|
| JWT — 3 части через точку | Нет точек или частей ≠ 3 | Ошибка под полем, push **не отправляем** |
| PEM — правильный заголовок | Не начинается с `-----BEGIN PRIVATE KEY-----` | Ошибка под полем, push **не отправляем** |
| terminal_id — непустой | Пустая строка или пробелы | Ошибка под полем, push **не отправляем** |

### 9.4. Генерация QR настройки

Кнопка «Сгенерировать QR» → iikoWeb формирует JSON (terminal_id, jwt, private_key, revision) → кодирует в Base64 → генерирует QR-изображение (PNG).

Модальное окно с QR: «Отсканируйте QR-код на кассовом терминале». Кнопки: **Распечатать**, **Скачать**, **Закрыть**.

## 10. Конфигурация плагина — config.xml

Все настройки плагина хранятся в файле `config.xml` в папке плагина на терминале.

### 10.1. Полная таблица параметров

| Параметр | Тип | По умолчанию | Описание | Кто заполняет |
|----------|-----|:------------:|----------|:-------------:|
| `BaseUrl` | string | `https://api.wbpay.ru` | Базовый URL WB Pay API. Sandbox: `https://sandbox.wbpay.ru` | Сборка/дистрибуция |
| `TerminalId` | string | *(пусто)* | ID терминала в системе WB Pay. Передаётся в теле каждого `/register` запроса | Канал A (Transport), B (QR) или C (вручную) |
| `JwtToken` | string | *(пусто)* | JWT Bearer Token для авторизации. TTL = 1 год. Передаётся в `Authorization` заголовке | Канал A, B или C |
| `PrivateKeyPath` | string | `private.pem` | Относительный путь к файлу ED25519 ключа. Используется для X-Signature | Канал A, B или C |
| `Country` | string | `RU` | Код страны ISO 3166-1 alpha-2. Передаётся в `X-Request-Country` | Сборка/дистрибуция |
| `Region` | string | `RU-MOW` | Код региона ISO 3166-2. Передаётся в `X-Request-Region` | Сборка/дистрибуция |
| `HttpTimeoutSec` | int | `10` | Таймаут одного HTTP-запроса в секундах | Сборка/дистрибуция |
| `PollingIntervalMs` | int | `3000` | Интервал между запросами polling в миллисекундах (рекомендовано 2000-3000) | Сборка/дистрибуция |
| `PollingTimeoutSec` | int | `120` | Максимальное время ожидания финального статуса в секундах | Сборка/дистрибуция |
| `RetryCount` | int | `3` | Количество повторных попыток при транзиентных ошибках | Сборка/дистрибуция |
| `RetryBaseIntervalMs` | int | `1000` | Базовый интервал exponential backoff в миллисекундах | Сборка/дистрибуция |
| `CredentialsRevision` | long | `0` | Revision для sync с iikoWeb (timestamp UTC мс). 0 = без revision | Автоматически (push/pull/QR) |

> **Важно:** параметры `BaseUrl`, `Country`, `Region`, таймауты и retry задаются при **сборке/дистрибуции** плагина и НЕ доставляются через Transport или QR. Через каналы A/B/C доставляются только `TerminalId`, `JwtToken`, `PrivateKeyPath` (файл `private.pem`) и `CredentialsRevision`.

### 10.2. Валидация при запуске

| Проверка | Условие ошибки | Результат |
|----------|---------------|-----------|
| config.xml существует | Файл отсутствует или ошибка парсинга | Логируем Error, «WB-кошелек» **не появляется** |
| `BaseUrl` заполнен и валиден | Пустой или невалидный URI | То же |
| `TerminalId` заполнен | Пустая строка или пробелы | То же |
| `JwtToken` заполнен | Пустая строка | То же |
| `private.pem` доступен | Файл не найден или нечитаем | То же |
| `Country` заполнен | Пустая строка | То же |
| `Region` заполнен | Пустая строка | То же |

Если **любая** проверка не прошла — платёжная система не регистрируется, тип оплаты не появляется на кассе.

## 11. Обработка ошибок и retry

### 11.1. Стратегия retry — Exponential Backoff

| Попытка | Интервал перед попыткой |
|:-------:|:-----------------------:|
| 1 (исходный запрос) | 0 сек (немедленно) |
| 2 | 1 сек |
| 3 | 2 сек |
| 4 | 4 сек |

Параметры из config.xml: `RetryCount = 3` (т.е. до 4 попыток: 1 исходная + 3 retry), `RetryBaseIntervalMs = 1000`.

### 11.2. Какие HTTP-коды ретраить

| Действие | HTTP-коды | Обоснование |
|----------|-----------|-------------|
| **Retry** | 408, 429, 500, 502, 503, 504 | Транзиентные ошибки сервера или инфраструктуры |
| **Не retry** | 400 | Ошибка в данных запроса — повторный вызов с теми же данными даст ту же ошибку |
| **Не retry** | 403 | Ошибка авторизации — нужно вмешательство администратора (см. 11.4) |

### 11.3. Идемпотентность

| Эндпоинт | Идемпотентен? | Поведение при retry |
|----------|:-------------:|---------------------|
| `/orders/do` | **Да** | Повторный вызов с тем же `order_id` безопасен, двойного списания не будет |
| `/refunds/do` | **Да** | Повторный вызов с тем же `refund_id` безопасен |
| `/orders/offline/register` | **Нет** | Повторный вызов создаёт **новый** `order_id`. Старый истечёт по TTL (2 мин) — это нормально |
| `/refunds/register` | **Нет** | Аналогично — новый `refund_id` |

### 11.4. Максимальное время при полном сетевом сбое

| Компонент | Расчёт | Значение |
|-----------|--------|:--------:|
| HTTP-запросы | `HttpTimeoutSec × (RetryCount + 1)` = 10 × 4 | 40 сек |
| Интервалы backoff | 0 + 1 + 2 + 4 | 7 сек |
| **Итого** | | **~47 сек** |

Через 47 секунд кассиру показывается: «Нет связи с сервером WB Pay».

### 11.5. HTTP 403 — ошибка авторизации

Особый случай. JWT невалиден — истёк (TTL 1 год), отозван или повреждён.

- **Retry запрещён** — повторные попытки не помогут
- Кассиру: «Ошибка авторизации WB Pay. Обратитесь к администратору для проверки настроек плагина»
- Логируем: HTTP-код, URL запроса, на уровне Error
- **Восстановление:** администратор обновляет JWT через iikoWeb (канал A), QR-проливку (канал B) или ручную правку config.xml (канал C)

> Формат тела ответа при HTTP 403 не документирован WB Pay API. Плагин обрабатывает этот код **исключительно по HTTP-статусу**, без привязки к телу ответа.

## 12. Логирование

| Событие | Уровень | Что логировать |
|---------|:-------:|----------------|
| Инициализация не удалась | Error | Причина ошибки, путь к config.xml |
| HTTP 403 на любом API-вызове | Error | URL запроса, HTTP-код |
| Все retry исчерпаны (сетевой сбой) | Error | URL, тип ошибки, количество попыток |
| Экстренный возврат FISCAL_ERROR — fail | Error | order_id, refund_id, fail_reason_code, fail_reason_description |
| QR-проливка: ошибка записи файла | Error | Путь к файлу, тип ошибки |
| Transport push: невалидный payload | Error | Тип ошибки, payload (**без секретов**) |
| Transport push: конфликт revision | Warning | Входящая revision, локальная revision |
| Transport pull: ошибка сети | Warning | Тип ошибки |
| Успешная QR-проливка | Info | terminal_id (**без JWT и private_key**) |
| Transport push: credentials обновлены | Info | terminal_id, revision (**без JWT и private_key**) |
| Transport push: credentials удалены | Info | terminal_id |
| Каждый HTTP-запрос к WB Pay API | Debug | HTTP-метод, URL, HTTP-код ответа, время (мс) |
| Polling: смена статуса | Debug | order_id / refund_id, новый статус |

> **Правило безопасности:** JWT-токен, private_key и полное тело запроса/ответа **не должны** логироваться на уровне Info и выше. Чувствительные данные — только на уровне Debug.

## 13. Фискализация

### 13.1. Тип оплаты

WB-кошелек регистрируется как фискальный тип оплаты **«Электронный»** (ФФД 1.05+). Фискальный чек обязателен для каждой операции.

### 13.2. Рекомендуемая настройка

`FiscalChequeBeforePaymentEnabled = true` — чек печатается **до** списания денег. Если чек не удалось напечатать — оплата просто не проводится. FISCAL_ERROR невозможен. Соответствует п. 5.11 ФЗ-54.

### 13.3. Комбинирование

Электронный тип оплаты **нельзя** комбинировать с нефискальными типами в одном чеке (ограничение ФФД 1.05+).

## 14. Scope — что входит и не входит

### MVP (первый релиз)

- ✅ QR-оплата через WB-кошелек
- ✅ Полные возвраты
- ✅ Фискализация (ФЗ-54, ФФД 1.05+)
- ✅ Онбординг: каналы A (Transport), B (QR), C (ручной)
- ✅ Страница администрирования в iikoWeb
- ✅ Обработка FISCAL_ERROR (EmergencyCancelPayment)
- ✅ Режимы: «Обслуживание столов» и «Фаст-фуд»

### MVP+ (следующий этап)

- ⏳ Касса самообслуживания (КСО)
- ⏳ Мобильные официанты
- ⏳ Маркетплейс (витрина)
- ⏳ Частичные возвраты

### Вне scope

- ❌ Лояльность («ягодки» WB) — обрабатывается на стороне WB
- ❌ Офлайн-режим — API WB Pay только онлайн, каждая операция требует обращения к серверу в реальном времени
- ❌ Токенизация (повторная оплата без QR) — недоступна в Offline API WB Pay
- ❌ Не-Россия — только RUB (код 643), только РФ
- ❌ Автоматическое обновление JWT — не предусмотрено API WB Pay

## 15. Ограничения, которые нужно помнить

| Ограничение | Значение | Контекст |
|-------------|----------|----------|
| TTL QR-кода гостя | ~5 минут | Потом нужно обновить в приложении WB |
| TTL `order_id` / `refund_id` | 2 минуты после `/register` | Если `/do` не вызван — идентификатор протухает |
| TTL JWT-токена | 1 год | Механизм авто-обновления отсутствует, обновление через iikoWeb |
| Polling таймаут | 120 сек (конфигурируется: `PollingTimeoutSec`) | Если статус не получен — операция отклоняется |
| Гарантия WB по финализации | до 24 часов | На стороне сервера WB; плагин ограничен своим таймаутом |
| Задержка `/register` → `/do` | < 5 сек (HTTP round-trip + ED25519) | TTL 2 мин более чем достаточен |
| Макс. позиций в заказе | 1000 | Ограничение массива `positions[]` |
| Макс. длина строковых полей | 2048 символов | `invoice_id`, `positions[].name` |
| Поддерживаемая версия SDK | iikoFront API V9 | Совместимость с другими версиями не гарантирована |
| Валюта | Только RUB (код 643) | Константа во всех запросах |
| Территория | Только Российская Федерация | `Country=RU`, `Region=RU-*` |
| Диалоги плагина | Только стандартные SDK | `ShowExtendedInputDialog`, `ShowInputDialog` — кастомный UI не реализуется |

### Критические (требуют ручного вмешательства)

| # | Ситуация | Действие | Сообщение |
|:-:|----------|----------|-----------|
| 1 | FISCAL_ERROR: экстренный возврат отклонён (`status = failed`) | Показать fail_reason + направить в поддержку WB Pay | Текст из `fail_reason_description` |

### Высокие (блокируют операцию)

| # | Ситуация | Действие | Retry? |
|:-:|----------|----------|:------:|
| 4 | HTTP 403 (JWT невалиден) | Направить к администратору | Нет |
| 5 | Сетевые ошибки (все retry исчерпаны) | Показать ошибку связи | Да (3 попытки) |

### Средние (отклоняют текущую операцию)

| # | Ситуация | Действие |

| 7 | QR просрочен / невалиден / дубликат | Отклонить оплату, показать причину |
| 8 | Недостаточно средств / лимит / нет способа оплаты | Отклонить оплату, показать причину |

| 10 | Возврат невозможен (`REFUND_NOT_POSSIBLE`) | Отклонить возврат, направить в поддержку WB Pay |
| 11 | Transport push с невалидным payload | Игнорировать, сохранить предыдущее состояние |
| 12 | QR-проливка: ошибка записи файлов | Показать ошибку, предложить повторить |

| # | Ситуация | Действие |
|:-:|----------|----------|
| 13 | Инициализация: credentials пусты | Не регистрировать «WB-кошелек» |
| 14 | Transport pull: ошибка сети | Использовать локальный config.xml |

## 18. Резюме — 7 ключевых тезисов

1. **WB Pay** — QR-оплата из кошелька Wildberries на наших POS-терминалах. Кассир сканирует QR гостя → деньги списываются → чек печатается.

2. **Плагин** реализует стандартный интерфейс `IExternalPaymentProcessor` (V9 SDK) — три метода: `Pay()`, `ReturnPayment()`, `EmergencyCancelPayment()`.

3. **Трёхшаговая модель**: register → do → poll status — одинаковая для оплаты и возврата. 6 эндпоинтов, polling-модель, JSON/HTTPS.

4. **Двойная авторизация**: JWT в `Authorization` (все запросы) + ED25519-подпись в `X-Signature` (только `/register`). Суммы в копейках, валюта = 643 (RUB).

5. **Онбординг**: три канала доставки credentials — Transport push/pull (основной), QR-проливка (fallback), ручной (аварийный). Revision-based sync для защиты от устаревших данных.

6. **FISCAL_ERROR**: критический сценарий с экстренным возвратом — требует особого внимания при реализации. Рекомендуем `FiscalChequeBeforePaymentEnabled = true` для исключения этого сценария.

7. **Прототип**: все сценарии можно попробовать на https://view21.ru/prototype/aurora — оплата, возврат, FISCAL_ERROR, настройка, админ-панель.

Спецификация полностью готова, прототип работает. Если есть вопросы по конкретным сценариям, ошибкам, полям API — всё описано выше. Код пишем по спецификации, прототип — для визуального понимания.
